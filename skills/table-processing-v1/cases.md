# 表处理 Skill V1 验收场景

## 场景 0：第一次打开失败更像路径格式问题

### 用户说法

“刚才打开失败了，你继续看这份台账。”

### Skill 应做什么

1. 先判断这更像路径格式问题，而不是文件内容问题
2. 先做 Windows 原生反斜杠路径纠偏
3. 再继续 `open_workbook` 或 `normalize_table`
4. 对应设计：`docs/plans/2026-03-22-path-recovery-skill-design.md`

### Skill 不应做什么

- 不应直接说文件坏了
- 不应跳过入口恢复直接开始分析

## 场景 0B：中文路径兼容问题

### 用户说法

“文件就在中文目录里，你继续处理。”

### Skill 应做什么

1. 先判断文件存在，但当前更像中文路径兼容问题
2. 如果宿主环境支持复制动作，先征求用户确认；只有用户同意后，才走 ASCII 临时副本
3. 再继续 `open_workbook` 读取 sheet 列表
4. 对应设计：`docs/plans/2026-03-22-path-recovery-skill-design.md`

### Skill 不应做什么

- 不应直接说文件不存在
- 不应还没恢复路径就猜哪张 sheet 是目标表

## 场景 1：用户只想先看表

### 用户说法

“帮我看看这个 Excel 里的 Sales 表。”

### Skill 应做什么

1. 调 `normalize_table`
2. 若表头明确，调 `preview_table`
3. 用自然语言总结前几行和列名
4. 对应模板：`requests.md` -> “单表模板 / 0. 先列出工作簿 sheet”、“1. 表头判断” 与 “2. 预览前几行”

### Skill 不应做什么

- 不应跳过表头检查
- 不应直接开始筛选或汇总

## 场景 2：用户要做单表汇总

### 用户说法

“按地区统计销售额。”

### Skill 应做什么

1. 确认目标表
2. 如有必要先 `preview_table` 或 `stat_summary`
3. 调 `group_and_aggregate`
4. 对应模板：`requests.md` -> “单表模板 / 3. 统计摘要” 与 “6. 分组汇总”

### Skill 不应做什么

- 不应自己先算一遍
- 不应自己猜聚合字段

## 场景 3：两张表是否该追加

### 用户说法

“这两张销售表帮我合一下。”

### Skill 应做什么

1. 调 `suggest_table_workflow`
2. 如果推荐 `append_tables`，问“是否先把 B 表追加到 A 表下方”
3. 用户确认后再调 `append_tables`
4. 对应模板：`requests.md` -> “双表模板 / 1. 双表工作流判断” 与 “3. 结构相同表追加”

### Skill 不应做什么

- 不应直接执行 `append_tables`
- 不应使用“同构”“schema”这类词

## 场景 4：两张表是否该关联

### 用户说法

“把客户表和订单表合到一起看。”

### Skill 应做什么

1. 调 `suggest_table_workflow`
2. 若推荐 `join_tables`，继续问关联列和保留方式
3. 用户确认后再调 `join_tables`
4. 对应模板：`requests.md` -> “双表模板 / 1. 双表工作流判断” 与 “4. 显性关联执行”

### Skill 不应做什么

- 不应直接说 `left join`
- 不应跳过保留方式确认

## 场景 5：多张表先追加再关联

### 用户说法

“先把两张销售明细合并，再和客户表对上。”

### Skill 应做什么

1. 调 `suggest_multi_table_plan`
2. 向用户解释第一步是先追加、第二步是再关联
3. 用返回的 `question` 字段逐步确认
4. 对应模板：`requests.md` -> “多表模板 / 1. 多表顺序计划”

### Skill 不应做什么

- 不应假装 `step_1_result` 自己就是真实结果；要等前一步执行后再把真实 `result_ref` 绑定回来
- 不应跳过计划说明，直接说“我已经全部做完了”
- 不应在没有前一步真实结果时私自伪造 `result_ref_bindings`

## 场景 6：表头不明确

### 用户说法

“先帮我清一下这张表。”

### Skill 应做什么

1. 调 `normalize_table`
2. 若返回 `needs_confirmation`，先让用户确认表头
3. 只有确认后才进入后续处理
4. 对应模板：`requests.md` -> “单表模板 / 1. 表头判断”

### Skill 不应做什么

- 不应继续做筛选、排序、分组

## 场景 7：用户问‘先怎么处理’

### 用户说法

“我也不知道先看什么，你帮我判断一下。”

### Skill 应做什么

1. 单表先 `stat_summary`
2. 双表先 `suggest_table_workflow`
3. 多表先 `suggest_multi_table_plan`
4. 对应模板：按表数量从 `requests.md` 里选“单表模板 / 3”、“双表模板 / 1”、“多表模板 / 1”

### Skill 不应做什么

- 不应把“判断步骤”放进 Skill 自己的自由发挥

## 场景 8：用户要求一步到位全自动多表执行

### 用户说法

“你直接全部做完，不要一步步问我。”

### Skill 应做什么

1. 先给出多表计划
2. 明确说明当前 V1 更适合先确认关键步骤
3. 不夸大当前多步自动执行能力
4. 对应模板：`requests.md` -> “多表模板 / 1. 多表顺序计划”

### Skill 不应做什么

- 不应虚构当前 Tool 层还没有的会话态中间结果执行能力

## 2026-03-23 兼容补充
- 兼容恢复成功后，后续案例应默认按“第几个 Sheet”继续，而不是再次依赖中文 Sheet 名。
- 只要还没有得到用户确认，就不能在案例里假设已经复制了 ASCII 临时副本。
