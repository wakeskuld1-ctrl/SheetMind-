# 分析建模 Skill V1 验收场景

## 场景 1：用户只想先判断这张表能不能建模

### 用户说法

“先帮我看看这张表适不适合建模。”

### Skill 应做什么

1. 先确认目标文件与 Sheet
2. 优先调 `analyze_table`
3. 用自然语言解释主要风险、整体健康度和建议下一步
4. 对应模板：`requests.md` -> “观察诊断模板 / 1. 默认健康诊断”

### Skill 不应做什么

- 不应一上来直接跑回归或聚类
- 不应自己编造“已经可以直接建模”的结论

## 场景 2：用户要先看统计摘要

### 用户说法

“先给我看一下这几个字段的统计情况。”

### Skill 应做什么

1. 确认字段范围
2. 调 `stat_summary`
3. 解释数值列、类别列、布尔列的关键观察
4. 对应模板：`requests.md` -> “观察诊断模板 / 3. 统计摘要” 与 “5. 带显式类型转换的统计摘要”

### Skill 不应做什么

- 不应把所有统计量一次性堆给用户
- 不应绕过 Tool 自己算均值或分布

## 场景 3：用户明确要做线性回归，但没有给目标列

### 用户说法

“帮我做个线性回归。”

### Skill 应做什么

1. 先追问“你想预测的是哪一列”
2. 再追问“你希望用哪些列来解释或预测它”
3. 确认缺失值处理方式
4. 关键参数齐全后再调 `linear_regression`
5. 对应模板：`requests.md` -> “线性回归模板 / 1. 标准线性回归”

### Skill 不应做什么

- 不应自己猜 `target`
- 不应自己猜 `features`
- 不应在缺参数时直接发请求

## 场景 4：用户明确要做线性回归，但字段类型可疑

### 用户说法

“我想预测销售额，用客户数、广告投入、折扣率试一下。”

### Skill 应做什么

1. 识别这已经接近可执行建模请求
2. 如字段看起来像文本数字，先确认是否按数值理解
3. 必要时补 `casts`
4. 再调 `linear_regression`
5. 对应模板：`requests.md` -> “线性回归模板 / 2. 带显式类型转换的线性回归”

### Skill 不应做什么

- 不应忽略明显的类型风险
- 不应在用户未确认时随意替列做类型假设

## 场景 5：用户明确要做逻辑回归，但没有说哪个值算正类

### 用户说法

“帮我判断客户会不会流失，做逻辑回归。”

### Skill 应做什么

1. 先确认目标列
2. 明确追问哪个值算最关心的结果，也就是 `positive_label`
3. 再确认特征列与缺失值策略
4. 必要时先用 `stat_summary` 看目标列分布
5. 对应模板：`requests.md` -> “逻辑回归模板 / 1. 标准逻辑回归” 与 “诊断后进入建模的模板串联 / 2. 先看统计摘要，再做逻辑回归”

### Skill 不应做什么

- 不应自己猜“yes”“1”“流失”就是正类
- 不应把训练准确率包装成最终业务承诺

## 场景 6：用户明确要做聚类，但没有说分几类

### 用户说法

“把这些客户分群看看。”

### Skill 应做什么

1. 先确认用于分群的特征列
2. 追问想先分成几类
3. 如果用户没有想法，可以建议首轮先试 3 类
4. 确认缺失值处理方式后调 `cluster_kmeans`
5. 对应模板：`requests.md` -> “聚类模板 / 1. 标准 KMeans 聚类” 与 “3. 首轮试探分组数”

### Skill 不应做什么

- 不应直接替用户拍板最优分组数
- 不应把建议值伪装成系统自动计算值

## 场景 7：用户说先诊断，再决定做什么模型

### 用户说法

“你先看看这张表的数据情况，再告诉我适合做什么分析模型。”

### Skill 应做什么

1. 先调 `analyze_table`
2. 必要时补 `stat_summary`
3. 用自然语言解释当前更适合回归、分类、聚类，还是先回表处理层清洗
4. 如果用户再确认某个模型，才进入建模模板
5. 对应模板：`requests.md` -> “观察诊断模板 / 1、3” 与 “诊断后进入建模的模板串联”

### Skill 不应做什么

- 不应把诊断阶段直接跳成建模执行阶段
- 不应在用户还没确认前自动选择模型跑完

## 场景 8：用户想先看基础列画像，不要复杂诊断

### 用户说法

“先简单看看这些列是什么类型、空值多不多。”

### Skill 应做什么

1. 调 `summarize_table`
2. 用更轻量的方式解释列画像
3. 如用户继续深入，再升级到 `analyze_table` 或 `stat_summary`
4. 对应模板：`requests.md` -> “观察诊断模板 / 4. 基础列画像”

### Skill 不应做什么

- 不应默认给出过重的诊断或建模结果
- 不应在用户没要时直接进入模型 Tool

## 场景 9：表头还没确认，用户却直接要求建模

### 用户说法

“直接给我做逻辑回归吧。”

### Skill 应做什么

1. 先确认当前表是否已经进入可分析状态
2. 如果还没有确认表头，先退回表处理层确认表头
3. 只有表结构明确后，才继续建模前追问
4. 对应模板：不直接发建模请求，先回表处理 Skill 流程

### Skill 不应做什么

- 不应跳过表头确认直接发 `linear_regression`、`logistic_regression` 或 `cluster_kmeans`

## 场景 10：用户要求一步到位自动决定模型与参数

### 用户说法

“你直接选个最好的模型帮我跑完，不用问我。”

### Skill 应做什么

1. 明确说明当前 V1 不做模型自动优选
2. 先提供观察诊断结果
3. 再给出可选的下一步模型建议
4. 引导用户确认目标列、特征列、正类标签或分组数
5. 对应模板：`requests.md` -> “观察诊断模板”

### Skill 不应做什么

- 不应伪装成已具备自动选模、自动调参、自动定正类、自动定分组数的能力
- 不应越过用户确认直接生成建模结论
