# 分析建模 Skill V1 模拟对话验收稿

这个文档用于人工验收 `analysis-modeling-v1`。

验收目标不是看 Skill 会不会“自由发挥”，而是看它是否稳定符合这 5 条：

1. 默认先诊断，再决定是否建模
2. 用户明确点名模型时，可以直达，但不能跳过最小前置校验
3. 关键参数不齐时，先追问，不猜测
4. 结果解释只基于 Tool 返回，不伪造计算结论
5. 不夸大当前 V1 尚未支持的自动选模、自动调参或自动决策能力

---

## 场景 1：用户只想先判断这张表能不能建模

### 用户

先帮我看看这张表适不适合建模。

### 期望 Skill 回复

当前理解：你现在还不是马上要跑某个模型，而是想先知道这张表的数据质量和结构，是否适合进入后续分析建模。  
下一步动作：我会先做一轮表级诊断，看看这张表有没有明显的缺失、重复、异常分布或标识列风险。  
对应请求：

```json
{
  "tool": "analyze_table",
  "args": {
    "path": "tests/fixtures/model-sales.xlsx",
    "sheet": "Sales",
    "top_k": 5
  }
}
```

### 验收关注点

- 是否优先进入 `analyze_table`
- 是否没有直接跳去回归或聚类
- 话术是否是业务表达，不是技术黑话

### 通过判定

- 明确说明“先诊断”
- 发出的是 `analyze_table`
- 没有编造“已经可以建模”的判断

### 失败判定

- 直接发 `linear_regression` / `logistic_regression` / `cluster_kmeans`
- 跳过诊断，直接给模型建议

## 场景 2：用户要先看统计摘要

### 用户

先给我看一下销售额、地区和会员标记这几个字段的统计情况。

### 期望 Skill 回复

当前理解：你想先看指定字段的统计摘要，了解数值分布和类别集中情况。  
下一步动作：我会先对这几个字段做统计摘要，再把关键观察用自然语言告诉你。  
对应请求：

```json
{
  "tool": "stat_summary",
  "args": {
    "path": "tests/fixtures/basic-sales.xlsx",
    "sheet": "Sales",
    "columns": ["sales", "region", "is_member"],
    "top_k": 5
  }
}
```

### 验收关注点

- 是否优先使用 `stat_summary`
- 是否没有把所有统计量一次性灌给用户
- 是否没有绕过 Tool 自己算均值、中位数等

### 通过判定

- 路由到 `stat_summary`
- Skill 明确说“先看统计摘要”

### 失败判定

- 路由到不相关 Tool
- Skill 自己口算摘要

## 场景 2A：上游已经确认表头，并交来了 `table_ref`

### 用户

这张表我前面已经确认过表头了，你直接继续分析就行。

### 期望 Skill 回复

当前理解：这张表已经在表处理层确认过结构，所以这一步不需要你再重复提供文件路径、Sheet 名或重新确认表头。  
下一步动作：我会直接复用上游给出的 `table_ref` 继续做分析诊断。  
对应请求：

```json
{
  "tool": "analyze_table",
  "args": {
    "table_ref": "table_1234567890",
    "top_k": 5
  }
}
```

### 验收关注点

- 是否优先走 `table_ref`
- 是否没有退回 `path + sheet`
- 是否没有要求重新确认表头

### 通过判定

- 明确说明“复用上游确认态”
- 发出的请求里使用 `table_ref`

### 失败判定

- 重新追问文件路径、Sheet
- 再次要求用户确认表头

## 场景 3：用户明确要做线性回归，但没有给目标列

### 用户

帮我做个线性回归。

### 期望 Skill 回复

当前理解：你已经明确想做线性回归，但目前还缺少最关键的建模信息。  
下一步问题：你想预测的是哪一列？另外，你希望用哪些列来解释或预测它？  
确认后动作：等目标列和特征列明确后，我再进入线性回归请求模板。

### 本轮期望 Tool 请求

本轮不发 Tool 请求。

### 验收关注点

- 是否先追问 `target`
- 是否先追问 `features`
- 是否没有在缺参数时直接发 `linear_regression`

### 通过判定

- Skill 先追问，不猜参数
- 明确说明“确认后再发请求”

### 失败判定

- 直接发线性回归请求
- 自己猜目标列或特征列

## 场景 4：用户明确要做线性回归，参数基本齐全

### 用户

我想预测销售额，用客户数、广告投入、折扣率做线性回归。

### 期望 Skill 回复

当前理解：你想预测 `sales_amount`，并用 `customer_count`、`ad_spend`、`discount_rate` 作为解释变量。  
下一步问题：如果这些列里有空值，你希望跳过这些记录，还是等我先帮你判断是否需要额外清洗？  
确认后动作：默认按最保守的方式先跳过缺失行，再调用线性回归。

### 若用户确认“先跳过缺失行”，期望 Tool 请求

```json
{
  "tool": "linear_regression",
  "args": {
    "path": "tests/fixtures/model-sales.xlsx",
    "sheet": "Sales",
    "features": ["customer_count", "ad_spend", "discount_rate"],
    "target": "sales_amount",
    "intercept": true,
    "missing_strategy": "drop_rows"
  }
}
```

### 验收关注点

- 是否保留最小前置校验
- 是否明确缺失值处理方式
- 是否没有暴露一堆底层参数给低 IT 用户

### 通过判定

- 发出的请求字段与模板一致
- `missing_strategy` 使用 `drop_rows`

### 失败判定

- 自行发明不存在的参数
- 跳过缺失值处理确认

## 场景 5：用户明确要做逻辑回归，但没有说哪个值算正类

### 用户

帮我判断客户会不会流失，做逻辑回归。

### 期望 Skill 回复

当前理解：你想做一个“是否流失”的二分类模型，但还需要先确认哪个结果算你最关心的正类。  
下一步问题：在目标列里，哪个值算你最关注的结果？比如是把“流失”当成正类，还是把“未流失”当成正类？  
确认后动作：我再继续确认特征列和缺失值处理方式，然后调用逻辑回归。

### 本轮期望 Tool 请求

本轮不发 Tool 请求。

### 验收关注点

- 是否追问 `positive_label`
- 是否没有自己猜“yes”“1”“流失”就是正类

### 通过判定

- Skill 明确追问正类定义

### 失败判定

- 直接发 `logistic_regression`
- Skill 替用户决定正类

### 失败判定

- 直接发逻辑回归请求

## 场景 6A：逻辑回归返回“目标列只有一个类别”

### 用户

那你直接帮我跑逻辑回归吧。

### 期望 Skill 回复

当前理解：这次没有跑通，不是因为系统不会做逻辑回归，而是因为你现在选的目标列只有一个类别，暂时不满足二分类建模前提。  
下一步建议：先看一下目标列分布，确认是不是目标列选错了；如果没选错，就需要换一个真正只有两类结果的目标列，再继续做逻辑回归。  
下一步动作：优先建议调用 `stat_summary` 查看目标列分布，而不是硬解释模型结果。

### 验收关注点

- 是否把失败解释成“数据前提问题”
- 是否明确建议先看目标列分布
- 是否没有伪造逻辑回归结果

### 通过判定

- 明确说明“只有一个类别”
- 给出“先看分布 / 更换目标列”的下一步建议

### 失败判定

- 把问题说成模型算法错误
- 继续输出不存在的分类结果或系数解读

## 场景 7：用户明确要做聚类，但没有说分几类

### 用户

把这些客户分群看看。

### 期望 Skill 回复

当前理解：你想做客户分群，但现在还缺一个关键决定，就是先分成几类更合适。  
下一步问题：你想先分成几类看看？如果你现在没有明确想法，首轮我建议可以先试 3 类。  
确认后动作：等分组数和特征列确认后，我再调用聚类。

### 本轮期望 Tool 请求

本轮不发 Tool 请求。

### 验收关注点

- 是否追问 `cluster_count`
- 是否区分“建议值”和“自动判定值”

### 通过判定

- Skill 把 3 类说成建议，不是系统自动最优

### 失败判定

- Skill 直接替用户决定分组数

## 场景 8：用户明确要做聚类，且字段类型需要确认

### 用户

我想按年消费、到店次数、退款率把客户分成 3 类。

### 期望 Skill 回复

当前理解：你已经给了比较完整的聚类目标，想用 `annual_spend`、`visit_count`、`refund_rate` 这三列把客户分成 3 类。  
下一步问题：如果这些列里有文本形式的数字，我会先按数值处理；如果你担心字段类型不稳定，我也可以先按显式数值列来处理。  
确认后动作：调用 KMeans 聚类。

### 若用户确认继续，期望 Tool 请求

```json
{
  "tool": "cluster_kmeans",
  "args": {
    "path": "tests/fixtures/model-segments.xlsx",
    "sheet": "Customers",
    "features": ["annual_spend", "visit_count", "refund_rate"],
    "cluster_count": 3,
    "max_iterations": 100,
    "missing_strategy": "drop_rows"
  }
}
```

### 验收关注点

- 是否参数齐全后再调用聚类
- 是否没有发明当前不存在的聚类配置字段

### 通过判定

- 请求与模板一致

### 失败判定

- 多写了不存在字段
- 少了关键字段

对应请求：

```json

    "path": "tests/fixtures/model-sales.xlsx",
    "sheet": "Sales",
    "top_k": 5
  }
```

### 验收关注点

- 是否把“先诊断再选模型”走完整

### 通过判定

- 先发 `analyze_table`
- Skill 只给“模型建议”，不直接执行

### 失败判定

- 自动替用户决定模型并直接执行

## 场景 11：表头还没确认，用户却直接要求建模

## 场景 12：用户要求一步到位自动决定模型和参数

### 用户

你直接选个最好的模型帮我跑完，不用问我。

### 期望 Skill 回复

当前理解：你希望尽量减少中间确认，直接得到一个可参考的分析方向。  
下一步说明：当前 V1 还不做自动选模、自动调参或自动决定正类/分组数；我可以先做观察诊断，再把适合的模型方向告诉你，然后你确认后我再执行。  
对应请求：

```json
{
  "tool": "analyze_table",
  "args": {
    "path": "tests/fixtures/model-sales.xlsx",
    "sheet": "Sales",
    "top_k": 5
  }
}
```

### 验收关注点

- 是否诚实暴露 V1 边界
- 是否没有伪装成具备自动选模能力

### 通过判定

- Skill 先诊断，再建议
- Skill 明确说当前不自动选模

### 失败判定

- 直接说“我帮你选好了最优模型”
- 直接执行且不追问关键参数
