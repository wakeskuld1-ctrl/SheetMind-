# 运维容量评估场景化交付说明

## 这份说明是给谁看的

- 运维人员
- 实施人员
- 需要做扩容、压测、活动保障、上线评审的人
- 手里数据不一定完整，但希望先拿到可执行判断的人

## 这个东西到底能解决什么问题

传统容量评估经常卡在三个地方：

- 业务侧只给了零散 Excel，数据不齐，没法直接建模
- 运维侧知道机器大概情况，但说不清当前实例到底够不够
- 项目时间紧，来不及做完整监控接入和长期趋势采样

这套交付能力解决的不是“做一张漂亮报表”这么简单，而是下面这类真实问题：

- 当前实例数是否还能扛住下一波业务增长
- CPU、内存、业务量哪个更可能先成为瓶颈
- 在高峰、活动、扩容评审前，应该先补哪些数据
- 数据不完整时，先给保守结论还是先补数，优先级怎么排
- 没有完整监控平台接入时，能不能先做一版可落地判断

## 它是用什么手段解决的

这套能力不是只做一个简单线性外推，而是把几种手段组合起来，尽量在不同数据条件下都能给出结果。

### 手段一：直接读 Excel 指标做容量判断

当用户已经有业务量、实例数、CPU、内存等表格数据时，系统会直接读取 Excel 中的指标，结合：

- 当前实例数
- 目标 CPU / 内存利用率
- 峰值放大系数
- 业务增长率
- 冗余要求
- 扩容步长

给出当前容量状态、建议实例数、主要瓶颈和下一步建议。

### 手段二：做弹性评估，不只靠单一线性关系

这次能力的重点不是“业务量乘一个固定系数”。

在分析时，会结合：

- 历史趋势方向
- 峰值模式
- 增长预期
- 冗余模式
- 饱和惩罚
- 服务等级和故障影响

形成更偏运维决策的弹性判断。也就是说，业务量和 CPU 的关系不要求永远是单一线性，数据充分时会吸收趋势证据，数据不足时则退化为保守评估和补数建议。

### 手段三：数据不全时，用受控 SSH 取证补事实

如果用户拿不出完整部署清单或指标表，也可以通过受限 SSH 补一些基础事实，例如：

- 当前机器有多少 CPU 核
- 总内存大概多少
- 进程快照里大致跑了几个服务实例
- 当前主机名和基础盘点信息

这里不是开放式远程执行，而是严格限制在只读白名单命令内，当前范围包括：

- `ps -ef`
- `top -b -n 1`
- `nproc`
- `free -m`
- `cat /proc/cpuinfo`
- `cat /proc/meminfo`
- `hostname`

这意味着它更适合“容量取证”和“评审辅助”，而不是远程运维操作。

### 手段四：就算证据弱，也给决策路径

有些场景下，用户只能提供：

- 服务名称
- 大概的服务等级
- 峰值类型
- 故障影响

这时系统仍然可以输出指导型结论，不会因为缺一堆列就直接停住。它会明确告诉用户：

- 现在能做出什么级别的判断
- 结论置信度为什么有限
- 下一步最该先补什么数据
- 补完之后应该如何再次评估

## 用户会在哪些场景下用到它

### 场景一：有 Excel 指标，想直接判断要不要扩容

用户常见问题：

- 活动前要不要扩容
- 当前实例数是否还能承压
- 如果增长 20%，需要加几台或加几个实例

可用材料：

- Excel 中的时间、实例数、CPU、内存、业务量
- 服务等级、SLA、峰值模式
- 当前部署实例数和冗余要求

系统处理方式：

- 直接读取 Excel 指标
- 结合增长、峰值和目标利用率做容量测算
- 输出建议实例数和瓶颈资源

最终交付结果：

- 一份可直接发给业务或运维评审的 Excel 报表
- 明确的“当前是否有风险”和“建议扩到多少”

### 场景二：只有部分指标，但知道服务画像和部署要求

用户常见问题：

- 没有完整监控，但要先做一版评审意见
- 只能拿到 CPU 或部分业务量，怎么先判断

可用材料：

- 不完整的 Excel 指标
- 服务等级、故障影响、峰值模式
- 当前实例数、扩容步长、冗余方式

系统处理方式：

- 用已有指标形成部分量化结论
- 用服务画像和部署策略补足保守判断
- 把缺失项和风险单独列出来

最终交付结果：

- 一份可用于预审的 partial 报表
- 既有当前判断，也有“哪些地方还不能下死结论”的说明

### 场景三：拿不到完整 Excel，但允许只读 SSH 取证

用户常见问题：

- 服务部署情况说不清
- 监控平台数据拉不出来
- 只能先上机器看一下进程和主机资源

可用材料：

- SSH 主机信息
- 白名单命令返回结果
- 服务匹配规则
- 基础服务画像

系统处理方式：

- 通过受限 SSH 收集主机和进程事实
- 识别可能的实例数
- 把这些事实映射成容量证据
- 再结合服务等级和峰值要求给出建议

最终交付结果：

- 一份基于 SSH 证据的 partial 或 guidance-only 报表
- 明确告诉用户“当前结论更多是基于主机事实还是实例识别”

### 场景四：数据很少，但必须先给决策思路

用户常见问题：

- 评审会马上开，数据来不及补齐
- 业务方只问“有风险吗，要不要先扩”
- 当前更需要决策顺序，而不是极其精确的数值

可用材料：

- 服务名称
- 服务等级
- 峰值模式
- 故障影响
- 少量背景事实

系统处理方式：

- 先形成指导型判断
- 给出缺失输入优先级
- 给出人工决策路径
- 明确推荐下一步

最终交付结果：

- 一份 guidance-only 报表
- 重点不是“精确算出多少实例”，而是“现在该先怎么决策”

## 最终交付给用户的是什么

最终交付物是一个 Excel 报表，不要求用户去读 JSON，也不要求用户自己再拼装结果。

固定会有 4 个工作表：

- `结论页`
  直接看当前容量状态、建议实例数、瓶颈资源、推荐下一步。
- `资源测算页`
  看 CPU、内存等资源维度是怎么得出判断的。
- `证据与风险页`
  看本次判断用了哪些证据、当前证据强弱如何、有哪些风险。
- `补数与行动页`
  看缺什么数据、先补什么、后面按什么路径继续判断。

## 用户拿到这份 Excel 后能怎么用

- 给活动前保障评审做扩容依据
- 给上线评审会做容量风险说明
- 给运维值班团队做保守决策参考
- 给业务方说明“为什么这次先建议补数，不直接拍板”
- 给后续压测或监控接入提供补数清单

## 这次交付的核心特点

- Excel 是主交付物，不是附属导出
- 数据完整时能做量化判断
- 数据不完整时也不会卡死
- 可以结合受控 SSH 补主机和实例事实
- 不只给数字，还给补数优先级和行动路径
- 适合活动保障、扩容预审、上线评审这类真实运维场景

## 当前边界

- 这不是全自动监控平台替代品，长期容量规划仍然建议接入更完整的历史指标
- SSH 只用于白名单只读取证，不做任意命令执行
- 当只有极少量背景信息时，结果会以 guidance-only 为主，重点在决策思路而不是精确数值
- 如果服务实例识别规则不明确，系统会保守处理，不会随意猜实例数

## 一句话给用户解释

如果你后面要把这个能力介绍给别人，可以直接说：

“你给我 Excel 指标也行，只给部分机器和服务信息也行，甚至数据不全也行。我会用表格分析、弹性评估和受控 SSH 取证，把结果整理成一份可直接交付的容量评估 Excel，并告诉你现在该怎么判断、缺什么数据、下一步先做什么。”
