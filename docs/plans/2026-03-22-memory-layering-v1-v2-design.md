# Memory Layering V1/V2 Design

## 背景

当前系统已经开始同时依赖两类“记忆”：

1. 框架提供的上下文 / 会话记忆
2. 产品自己维护的本地运行时记忆

如果不尽早把这两类记忆的边界定义清楚，后面会出现三个典型问题：

- 同一个状态既写在框架记忆里，又写在本地数据库里，最后不知道该信谁
- 模型记住了“上一轮大概说过什么”，但系统并不知道“当前激活的 `table_ref` 到底是哪一个”
- 切模型、切线程、切入口后，对话还能继续，但产品状态已经漂移

因此需要一份单独的分层设计文档，明确：

- 框架记忆做什么
- 本地产品记忆做什么
- 谁是事实源
- 冲突时以谁为准
- V1 先做到哪里，V2 再增强什么

## 设计目标

- 允许接入 `codex`、`opencode`、`openclaw` 等具备上下文或记忆能力的框架
- 不让框架记忆替代产品自己的确定性状态层
- 让总入口 Skill、三层 Skill、Rust Tool 都有一致的状态读取原则
- 保持“Skill 不做计算，Tool 做计算，runtime 存状态”的总边界不变
- 为 V2 的多会话、多 agent、结果血缘、本地偏好演进留接口

## 非目标

- 不讨论具体某一家框架的内部 Prompt 细节
- 不把框架会话摘要复制进本地数据库做全文镜像
- 不在 V1 做向量记忆、语义召回或长期知识库
- 不在 V1 解决跨设备同步

## 核心判断

结论很明确：

- 有框架记忆，仍然需要本地产品记忆
- 但两者职责必须严格分层

一句话定义：

- 框架记忆负责“助手怎么理解用户”
- 本地产品记忆负责“系统当前真实状态是什么”

## 三层记忆架构

建议把整个系统的记忆拆成三层。

### 第 0 层：请求上下文层

这是每一轮请求天然携带的即时信息，例如：

- 当前用户输入
- 当前附件或当前文件路径
- 当前线程中的上一两轮对话

特点：

- 生命周期最短
- 不需要持久化
- 适合作为本轮即时判断依据

它的定位不是“记忆”，而是本轮输入。

### 第 1 层：框架会话记忆层

这是 `codex`、`opencode`、`openclaw` 一类框架可能提供的会话摘要、长期上下文、用户偏好记忆。

适合存：

- 用户偏好什么表达方式
- 用户是否喜欢“双层输出”还是“简洁输出”
- 用户不喜欢 `left join` 这种术语，要改用自然语言解释
- 用户通常先看质量诊断，再决定是否建模
- 用户当前更关心业务解释还是技术细节

不适合存：

- `active_table_ref`
- `schema_status`
- `current_stage`
- `current_workbook`
- `current_sheet`
- 确定性的建模参数主状态

原因很简单：

- 这些字段一旦错了，后续 Tool 链路就会串错
- 框架记忆更适合“偏好”和“语气”，不适合“事务状态”

### 第 2 层：本地产品记忆层

这是 `local-memory-runtime-v1` 对应的 SQLite 运行时。

它必须承担确定性状态存储职责。

适合存：

- `session_state`
- `active_table_ref`
- `schema_status`
- `current_workbook`
- `current_sheet`
- `current_stage`
- `selected_columns`
- `model_context`
- `event_logs`
- `table_refs`

它的特点是：

- 本地
- 可审计
- 可查询
- 可跨请求复用
- 可被 Tool 严格消费

## 谁是事实源

这个问题必须在 V1 就定死。

### 原则 1：产品状态以本地 runtime 为准

只要字段属于以下类别，就必须以 SQLite runtime 为唯一事实源：

- 当前处于哪一层
- 当前工作簿 / sheet 是什么
- 当前是否已经确认表头
- 当前 `table_ref` 是什么
- 当前模型上下文是什么

框架记忆即使“记得”这些内容，也只能作为候选参考，不能覆盖 runtime。

### 原则 2：用户偏好以框架记忆优先

以下内容可以优先放在框架记忆里：

- 偏好的输出风格
- 偏好的确认风格
- 是否喜欢先问后做
- 是否更关注业务统计而不是模型细节

后续如果需要产品级稳定配置，也可以镜像到本地 `user_preferences`。

### 原则 3：本轮输入优先级最高，但只影响当前轮

如果用户本轮明确说：

- “不要继续用上次那张表”
- “换成另一个 sheet”
- “这轮不要复用之前的模型设定”

那么本轮输入优先，并应显式写回本地 runtime。

## 记忆分层表

| 信息类型 | 推荐存放层 | 是否需要落本地 runtime | 说明 |
|----------|------------|------------------------|------|
| 当前 Excel 文件路径 | 本地产品记忆 | 是 | 属于运行时真状态 |
| 当前 sheet | 本地产品记忆 | 是 | 属于运行时真状态 |
| `active_table_ref` | 本地产品记忆 | 是 | 是跨层主句柄 |
| `schema_status` | 本地产品记忆 | 是 | 直接影响后续 Tool 是否可执行 |
| `current_stage` | 本地产品记忆 | 是 | 直接影响 orchestrator 路由 |
| `selected_columns` | 本地产品记忆 | 是 | 属于当前任务上下文 |
| `model_context` | 本地产品记忆 | 是 | 建模复用必须稳定 |
| 用户喜欢简洁回答 | 框架记忆 | 否 | 偏好类信息 |
| 用户不懂技术术语 | 框架记忆 | 否 | 表达层偏好 |
| 用户习惯先看统计再建模 | 框架记忆 | 可选 | 可记在偏好层，也可后续镜像 |
| 本轮用户问题原文 | 请求上下文 | 否 | 本轮即时输入，不必全文持久化 |

## 读写规则

### orchestrator 读取规则

总入口在每轮开始时遵循：

1. 先读请求上下文
2. 再读 `get_session_state`
3. 必要时参考框架记忆中的“偏好”
4. 最后决定走哪层 Skill

也就是说：

- 路由判断的主状态来自本地 runtime
- 表达方式和追问风格可以参考框架记忆

### Skill 写入规则

Skill 本身不保存状态，也不做计算。

它只负责：

- 触发 `update_session_state`
- 或调用那些会自动同步状态的 Tool

Skill 文档里定义的是“协议”，不是“存储本体”。

### Tool 写入规则

关键 Tool 自动写状态：

- `apply_header_schema`
  - 激活 `table_ref`
  - 把阶段写为 `table_processing`
  - 把 `schema_status` 写为 `confirmed`
- `stat_summary` / `summarize_table` / `analyze_table`
  - 推进到 `analysis_modeling`
- `linear_regression` / `logistic_regression` / `cluster_kmeans`
  - 推进建模上下文
  - 写事件日志
- `decision_assistant`
  - 推进到 `decision_assistant`

### 框架记忆写入规则

框架记忆只建议写：

- 输出风格偏好
- 追问耐受度
- 术语偏好
- 决策偏好

不要把 `table_ref`、`current_stage` 这类硬状态直接交给框架记忆托管。

## 冲突仲裁规则

未来只要出现“框架记忆”和“本地 runtime”不一致，统一按下面规则处理。

### 冲突 1：框架记忆说有 `table_ref`，runtime 没有

以 runtime 为准。

处理方式：

- 不直接复用框架记忆中的 `table_ref`
- 回到表处理层或要求重新确认

### 冲突 2：框架记忆说当前在分析阶段，runtime 说在表处理阶段

以 runtime 为准。

处理方式：

- 先判断是不是用户本轮明确要求切阶段
- 如果没有明确输入，按 runtime 当前阶段继续

### 冲突 3：用户本轮输入与 runtime 不一致

以用户本轮输入为准，但要显式更新 runtime。

例如：

- runtime 记得当前文件是 A
- 用户这轮说“换成 B 表”

这时应：

1. 接受用户本轮输入
2. 更新 `session_state`
3. 后续全链路改用 B

## V1 设计

### V1 目标

V1 解决的是“确定性运行时状态独立出来”。

当前 V1 应达成：

- 本地 SQLite runtime 已存在
- `session_state` 已能跨请求复用
- `table_ref` 已能落地并镜像到 runtime
- orchestrator 已明确“先读 session，再决定路由”
- 分析建模与决策层关键 Tool 已能自动推进状态

### V1 中框架记忆的定位

V1 对 `codex`、`opencode`、`openclaw` 等框架的使用建议是：

- 可以用
- 但只用于辅助交互
- 不进入确定性状态主链路

也就是：

- 有框架记忆更顺手
- 没框架记忆系统也不会影响产品主链路正确性

### V1 中不做的事

- 不做语义记忆召回
- 不做多 agent 共享状态仲裁
- 不做结果全文入库
- 不做偏好学习自动写回

## V2 设计方向

V2 再扩展，不要一次做重。

### V2-1：补齐产品级上下文

新增或补强：

- `model_contexts`
- `user_preferences`
- 更细的 `event_logs`
- 结果血缘摘要

这样可以让回归、聚类、决策链路复用更稳定。

### V2-2：支持框架记忆与本地 runtime 协作

这时可以正式定义：

- 偏好由框架记忆主存
- 关键偏好镜像到 `user_preferences`
- runtime 输出结构化摘要给框架记忆，用于自然语言续聊

这里仍然坚持：

- 框架记忆不替代 runtime
- 只是消费 runtime 的摘要

### V2-3：支持多会话 / 多 agent

如果后续会出现：

- 多个 Skill 并行
- 多 agent 协作
- 同一工作簿的多分支分析

那么 V2 需要新增：

- 更明确的 `session_id`
- `parent_session_id`
- `result_ref`
- 分支上下文
- 更明确的事件与血缘模型

## 对接 `codex` / `opencode` / `openclaw` 的统一建议

无论具体框架是谁，都建议遵循同一条原则：

### 可以交给框架的

- 用户偏好
- 风格偏好
- 术语偏好
- 最近讨论的话题摘要

### 不要交给框架的

- `table_ref`
- `schema_status`
- `current_stage`
- `current_workbook`
- `current_sheet`
- 确定性的建模参数主状态

### 最佳协作方式

框架每轮工作流程建议是：

1. 读取用户输入
2. 调 `get_session_state`
3. 结合框架自己的偏好记忆组织回复方式
4. 调用 Tool
5. 必要时显式 `update_session_state`

## 风险与控制

### 风险 1：把框架记忆当数据库用

控制方式：

- 明确只把框架记忆用于“偏好”和“表达”
- 产品状态必须有独立 runtime

### 风险 2：本地 runtime 和框架记忆双写同一字段

控制方式：

- 硬状态只写 runtime
- 偏好状态才允许写框架记忆

### 风险 3：V2 过早做复杂语义记忆

控制方式：

- 先把结构化状态做好
- 再谈语义层增强

## 最终判断

对这个 Excel Skill 产品来说：

- 框架有上下文和记忆系统，当然值得利用
- 但本地产品记忆仍然必须存在
- 两者不是替代关系，而是分层协作关系

最终推荐架构是：

1. 请求上下文层：承接本轮输入
2. 框架记忆层：承接偏好与表达习惯
3. 本地产品记忆层：承接确定性状态与运行时真相

这套边界一旦定住，后续无论接 `codex`、`opencode` 还是 `openclaw`，都不会把系统状态做乱。
