# Analyze Observation Enhancement Design

## 背景

当前 `analyze_table` 已经可以输出：

- 质量诊断：`structured_findings`
- 轻量业务观察：`business_observations`
- 面向用户的摘要：`human_summary`

现有观察主要覆盖主类别、数值范围、主维度和中心值，但对实际 Excel 场景里最常见、也最有业务解释力的三类字段仍不够友好：

- 日期列
- 时间列
- 金额列

用户已经明确要求先增强“观察能力”，再继续推进分析建模层。因此本轮目标是：在不增加 Skill 计算职责的前提下，在 Rust Tool 层补充这些字段的保守型业务观察。

## 目标

增强 `analyze_table` 的 `business_observations`，新增对日期、时间、金额三类字段的轻量观察，满足以下目标：

- 输出对非 IT 用户友好的直白中文观察
- 保持机器可消费的结构化 `type` + `column` + `message`
- 避免激进推断，优先保守识别
- 让后续决策助手和分析建模层可复用这些观察信号

## 非目标

V1 不做以下能力：

- 时间序列建模
- 节假日、工作日等复杂日历推断
- 币种自动识别
- 复杂金额清洗
- 行业知识型业务解释

## 方案对比

### 方案 A：继续把规则直接堆进 `analyze.rs`

优点：

- 改动少
- 实现快

缺点：

- `analyze.rs` 会越来越重
- 列语义识别、统计计算、文案输出混在一起

### 方案 B：新增轻量语义识别层

这是本轮已确认采用的方案。

优点：

- 职责清晰
- 未来回归、聚类、决策助手都可复用列语义判断
- 更符合 SRP / OCP

缺点：

- 比直接堆规则多一层组织成本

### 方案 C：把观察增强并入 `stat_summary`

优点：

- 统计语义更统一

缺点：

- `stat_summary` 会过早膨胀
- 本轮目标是增强 `analyze_table` 观察，而非重构统计桥接层

## 设计原则

### 1. 保守识别，不乱猜

列名和样本值必须同时满足一定条件，才判为日期、时间或金额列。

### 2. 观察优先于推理

输出的是“观察”，不是“业务结论”：

- 日期覆盖范围
- 数据是否集中在某个时间段
- 金额是否存在长尾或负值

### 3. 与现有观察互补，不重复轰炸

新增观察要尽量避免和现有：

- `numeric_range`
- `numeric_center`
- `median_center`
- `dominant_dimension`

产生大量重复信息。

## 架构

### 轻量语义识别层

新增内部语义识别结构，用于判断一列更像：

- `date`
- `time`
- `amount`

语义识别基于：

- 列名模式
- 列类型
- 样本值解析成功率

### 观察生成层

在现有 `build_business_observations(...)` 中新增三组候选观察：

- 日期观察
- 时间观察
- 金额观察

再继续走已有的 `push_business_observations(...)` 门禁和去重逻辑。

## 列语义识别规则

### 日期列

需要同时满足：

- 列名命中：`日期` / `date` / `day`
- 样本值中有足够比例可解析为日期

V1 支持的基础格式：

- `YYYY-MM-DD`
- `YYYY/MM/DD`
- `YYYY-MM-DD HH:MM:SS`（只取日期部分）

### 时间列

需要同时满足：

- 列名命中：`时间` / `time` / `hour`
- 样本值中有足够比例可解析为时间

V1 支持的基础格式：

- `HH:MM`
- `HH:MM:SS`
- `YYYY-MM-DD HH:MM:SS`（取时分秒部分）

### 金额列

需要同时满足：

- 列名命中：`金额`、`amount`、`amt`、`price`、`fee`、`total`、`实付`、`应收`、`收入`、`支出`
- 列本身是数值列，或在调用前已经通过 `casts` 转成数值列

## 新增观察类型

### 日期类

- `date_range`
  - 例：`订单日期` 列覆盖范围大致从 `2026-03-01` 到 `2026-03-31`
- `date_concentration`
  - 只在明显集中时触发
  - 例：`订单日期` 大部分记录集中在 `2026-03`

### 时间类

- `time_peak_period`
  - 例：`下单时间` 主要集中在下午
- `time_business_hour_pattern`
  - 只在明显集中时触发
  - 例：`下单时间` 大多发生在工作时段

### 金额类

- `amount_typical_band`
  - 例：`实付金额` 的常见区间大致在 `20` 到 `80`
- `amount_negative_presence`
  - 例：`净额` 列存在负数记录
- `amount_skew_hint`
  - 例：`实付金额` 少量高金额记录会明显拉高平均值

## Human Summary 处理

`human_summary.quick_insights` 继续复用 `business_observations`，但新增观察后要注意：

- 保持 2 条左右
- 优先展示更有业务解释力的观察
- 避免同时展示两个意思几乎相同的数值观察

## 测试策略

坚持 TDD：

1. 先写内存层失败测试，锁定日期/时间/金额观察类型
2. 再写 CLI 层失败测试，锁定 JSON 输出契约
3. 再做最小实现
4. 跑全量验证

建议首批测试覆盖：

- 日期列范围与月份集中
- 时间列上午/下午集中
- 金额列典型区间
- 金额列负数记录
- 金额列长尾提示
