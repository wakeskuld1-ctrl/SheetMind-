# Excel 风控优化器稳交付版设计

## 背景
- 当前输入表本质上是一个“赔率固定、返水固定、按号码测算单点盈亏暴露”的 Excel。
- 用户新需求已经从“测算表”升级为“受约束的整数优化交付物”：
  - 只能下调每个号码下注额，不能上调
  - 每个号码的新下注额必须保持整数
  - 最大亏损值必须不超过用户给定阈值
  - 在满足硬约束前提下，总退款金额最少
  - 在同等最优解中，让亏损号码数量尽量接近目标值，例如 19
- 交付形态已经批准为稳交付版：
  - 一个业务可直接使用的 `xlsm`
  - 一个同目录的 Rust `solver.exe`
  - Excel 内保留按钮，但按钮只做“调用求解器”的薄壳，不承载核心算法

## 已批准契约

### 业务规则
- 赔率固定按 `47`
- 返水固定按 `2%`
- `可赔付本金 = 总下注额 - 总下注额 * 2%`
- `单号码盈亏值 = 单号码赔付额 - 可赔付本金`
- `单号码赔付额 = 单号码下注额 * 47`
- `亏损号码` 定义为 `盈亏值 > 0`

### 优化规则
- 决策变量：每个号码调整后下注额
- 取值约束：
  - 新下注额必须是整数
  - `0 <= 新下注额 <= 原下注额`
- 主目标：
  - `总退款金额最少`
  - 等价于 `调整后总下注额最大`
- 次目标：
  - `|亏损号码数量 - 目标亏损号码数|` 最小
- 硬约束：
  - `最大亏损值 <= 用户输入阈值`

### 页面契约
- 第一页：`当前盘面`
  - 保留原业务布局和输入方式
  - 增加目标输入区
  - 增加“测算/生成建议”按钮
  - 保留当前状态测算摘要
- 第二页：`优化建议`
  - 展示优化摘要
  - 展示一段建议说明
  - 展示调整明细表

## 2026-04-20 页面还原批准补充

### 已批准页面方案
- 用户已批准采用“方案 A：原表 1:1 还原优先”。
- 页面目标不是做一张新的程序表，而是在保持现有 Rust 求解链路不变的前提下，把 Excel 壳恢复成接近原始 `计算器s6.1` 的业务页面。

### 第一页页面契约
- 第一页工作表名称改为更贴近原表的 `计算器`。
- 第一页必须保留原始横向分组盘面：
  - `双/单` 分区
  - 生肖标题
  - 每组 `号码 / 特下注额 / 赔付额 / 亏损额 / 建议修改`
  - 底部 `总下注额度 / 返水 / 可赔付本金`
  - 右侧当前风险摘要区
- 原始号码分布和读取坐标以原始模板为准，不再使用“01-49 纵向列表”的程序表布局。

### 必填字段标注契约
- 所有需要业务手动填写的字段，必须在字段名里显式标注 `（需要填写）`。
- 第一页至少需要标注的手填字段包括：
  - 各分组下注列标题：`特下注额（需要填写）`
  - `最大亏损目标值（需要填写）`
  - `目标亏损号码数（需要填写）`
- 非手填字段不得误标，避免现场误解。

### 第一页目标输入与操作区契约
- 目标输入区放在第一页右侧空白区，优先使用原表右下区域，避免破坏原盘面主体。
- 该区域必须与原表风格保持一致，包括：
  - 使用相近的底色、边框、居中样式
  - 作为业务页面的一部分，而不是外挂面板
- 该区域至少包含：
  - `最大亏损目标值（需要填写）`
  - `目标亏损号码数（需要填写）`
  - `测算并生成建议` 按钮
  - `求解状态`

### 第二页页面契约
- 第二页继续命名为 `优化建议`。
- 第二页不允许只是朴素结果表，必须继承第一页的业务风格。
- 第二页至少包含：
  - 优化摘要区
  - 建议说明区
  - 调整明细区
- 调整明细区必须明确展示：
  - 号码
  - 原下注额
  - 调整后下注额
  - 退款额
  - 调整后盈亏额
  - 是否建议下调

### 兼容性约束
- 本次页面还原仅改变工作簿展示契约与读取坐标，不改变已批准的业务规则、优化目标和整数约束。
- Rust 求解核心、退款规则和“只允许下调当前风险号码”规则保持不变。

## 交付架构

### 目标交付物
- `计算器-稳交付版.xlsm`
- `betting_solver.exe`

### 运行链路
1. 业务人员在第一页录入当前下注额，并填写目标值：
   - 最大亏损阈值
   - 目标亏损号码数
2. 点击第一页按钮
3. 按钮调用同目录 `betting_solver.exe`
4. `betting_solver.exe` 读取当前工作簿输入
5. Rust 求解器计算最优整数方案
6. 求解结果写回第二页 `优化建议`
7. 第一页摘要可继续保留“当前态”，第二页专门展示“建议态”

### Excel 与 Rust 分工
- Excel：
  - 页面承载
  - 输入体验
  - 按钮入口
  - 当前态公式展示
- Rust：
  - 读取第一页业务输入
  - 建模与整数优化求解
  - 生成第二页结果数据
  - 输出建议文字和明细表

## 技术设计

### 1. Workbook 层
- 第一页继续沿用现有号码布局，降低现场切换成本。
- 在第一页右侧或底部增加目标输入区：
  - `最大亏损目标值`
  - `目标亏损号码数`
  - `求解状态/最近一次求解时间`
- 第二页固定 3 个区域：
  - 摘要区
  - 建议说明区
  - 调整明细区

### 2. Rust 求解层
- 定义标准输入对象：
  - 原始号码列表
  - 原始下注额
  - 赔率
  - 返水比例
  - 最大亏损阈值
  - 目标亏损号码数
- 定义标准输出对象：
  - 每个号码原下注额
  - 每个号码新下注额
  - 每个号码退款额
  - 每个号码调整后盈亏值
  - 总退款金额
  - 调整后最大亏损值
  - 调整后亏损号码数
  - 建议说明文案

### 3. 最优性实现
- 采用分层目标而不是加权混合目标，避免权重调参污染业务语义：
  1. 第一层最大化 `保留总下注额`
  2. 第二层最小化 `|亏损号码数 - 目标值|`
- 由于下注额必须为整数，因此求解器必须支持整数变量。

### 4. 建议文案生成
- 文案不做自由生成，保持规则化模板，便于追溯。
- 建议说明至少覆盖：
  - 当前最大亏损与目标差距
  - 本次总退款金额
  - 调整后最大亏损值
  - 调整后亏损号码数量
  - 重点下调号码

示例风格：

```text
当前方案最大亏损为 2391.2，高于目标 1500。
在仅允许下调且保持整数下注额的前提下，已求得退款最少的可行方案。
本次共建议退款 640，调整后最大亏损降至 1498，亏损号码数量调整为 19。
重点建议下调的号码为：02、17、29、38。
```

## 关键技术决策

### 决策 1：稳交付版使用 `xlsm + exe`
- 原因：
  - 纯 VBA 难调试、难测试、难稳定交付
  - 核心算法属于整数优化，更适合 Rust 承担
- 影响：
  - 交付件从单文件变为双文件
  - 现场使用成本仍可接受

### 决策 2：VBA 只保留最薄调用层
- 原因：
  - 避免把业务逻辑再次埋进宏
  - 让后续维护主入口留在 Rust
- 影响：
  - 宏只做按钮触发、路径拼接、调用状态反馈

### 决策 3：第一页保留现场还原，第二页只放建议态
- 原因：
  - 现场操作路径最短
  - 不让业务人员在两个页面来回录入

### 决策 4：先做技术刺探验证 `xlsm` 回写方式
- 原因：
  - 当前仓库擅长读 Excel 与新建 xlsx，但“保留宏的 xlsm 原地回写”是否稳定，必须先验证
- 期望优先路径：
  - Rust 直接更新同一个 `xlsm`
- 允许的受控回退：
  - Rust 产出结构化中间结果，由薄 VBA 回填第二页
- 约束：
  - 无论底层采用哪种回写方式，对业务侧都保持“点按钮 -> 同一工作簿第二页出结果”的体验不变

## 风险与边界

### 已知风险
- `xlsm` 宏保留回写路径需要技术刺探，不能直接假设仓库现有导出链可用。
- 如果求解器依赖带来打包体积或运行时限制，需要在计划中单独收口。
- 如果当前原始 Excel 布局后续经常人工改动，第一页解析逻辑要基于明确坐标契约，而不是依赖模糊识别。

### 非目标
- 暂不做任意赔率、任意布局的通用求解平台
- 暂不做多模板兼容
- 暂不做云端或服务化部署
- 暂不做“允许上调下注额”的再分配模型

## 验收口径
- 求解结果满足：
  - 每个号码调整后下注额均为整数
  - 每个号码调整后下注额不高于原下注额
  - 最大亏损值不超过目标阈值
- 在所有满足硬约束的方案中：
  - 总退款金额最少
  - 若存在多个同等最优解，则亏损号码数量最接近目标值
- Excel 交付物满足：
  - 第一页可录入
  - 第一页可点击按钮
  - 第二页可读、可追溯
  - 交付件可在目标现场以 `xlsm + exe` 方式运行
