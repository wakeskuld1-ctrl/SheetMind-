# Path Recovery Skill Design

## 背景

当前 Excel 产品已经进入“总入口 Skill -> 子 Skill -> Rust Tool”的三层结构，但文件入口恢复这一步还不够稳定。

这轮要补齐两类最常见、也最容易被误判的问题：

1. Windows 路径格式错误
2. 中文路径在底层 Excel 读取链路中的兼容失败

这两个问题都属于“入口恢复”，不属于 Excel 内容分析，也不属于表处理、统计或建模计算本身。

本设计稿的目标，是让 `excel-orchestrator-v1` 和 `table-processing-v1` 在面对入口失败时，先做正确分诊，再决定后续 Tool 调用与用户话术。

## 目标

- 避免第一次打开失败就直接判断为“文件坏了”
- 给用户稳定、可解释、非技术化的话术
- 让总入口和表处理层遵守同一套恢复顺序
- 兼顾 Excel 低 IT 用户的理解成本
- 为后续真正的 Tool 层自动恢复预留边界

## 非目标

- 本轮不修改 Rust 计算 Tool
- 本轮不承诺已经自动修复所有 Excel 路径问题
- 本轮不在 Skill 层伪造文件复制或读取成功
- 本轮不把入口恢复写成“底层库已经修复”

## 两类失败场景

### 场景 A：Windows 路径格式问题

典型信号：

- 第一次打开工作簿就失败
- 错误更像路径语法不正确
- 旧会话状态还停留在其他测试文件上

Skill 该怎么理解：

- 当前失败更像路径格式问题
- 先纠正路径表达方式
- 改用 Windows 原生反斜杠路径重试

### 场景 B：中文路径兼容问题

典型信号：

- PowerShell 能正常定位到文件
- 文件确实存在
- 但 Excel Tool 链路直接读取中文路径失败
- 当前更像底层兼容问题，而不是文件不存在

Skill 该怎么理解：

- 当前失败更像中文路径兼容问题
- 如果宿主环境支持复制动作，可以先复制出一份只用于分析的 ASCII 临时副本
- 后续基于 ASCII 临时副本继续读取工作簿、列出 Sheet、判断目标表

## 恢复顺序

### 第一步：先判断是不是路径格式问题

如果错误更像 Windows 路径语法不正确，应该先做：

1. 明确告诉用户这更像路径格式问题
2. 不要先判断文件内容有问题
3. 改用 Windows 原生反斜杠路径继续重试

### 第二步：再判断是不是中文路径兼容问题

如果满足以下条件：

- 文件存在
- PowerShell 能定位到文件
- 但 Tool 层直接读中文路径失败

则应：

1. 明确告诉用户文件存在
2. 明确告诉用户失败点更像中文路径兼容
3. 如果宿主环境支持复制动作，先复制到 ASCII 临时路径
4. 用 ASCII 临时副本继续读取工作簿
5. 明确说明临时副本只用于分析，不代表修改原文件

### 第三步：最后才判断文件当前不可读

只有在前两步都不成立，或都已尝试但仍失败时，才把问题上升为：

- 文件当前不可读
- 需要用户重新确认路径或文件来源
- 暂时不能继续表处理

## Skill 落点

### `excel-orchestrator-v1`

总入口层需要做到：

- 先把问题分类为入口恢复，而不是内容分析
- 明确当前失败属于哪一类
- 路由回 `table-processing-v1`

它不应该：

- 直接猜哪张 Sheet 是目标表
- 直接开始统计摘要或建模
- 把路径问题说成文件损坏

### `table-processing-v1`

表处理层需要做到：

- 在单表入口前先做路径恢复分诊
- 支持“路径格式纠偏”与“ASCII 临时副本降级”两类动作说明
- 只有入口恢复完成后，才继续 `open_workbook`、`normalize_table`、`apply_header_schema`

它不应该：

- 还没恢复路径就直接猜目标 Sheet
- 跳过入口恢复直接进入统计或建模
- 把临时副本表述为“已经替换原文件”

## 用户话术原则

### 原则 1：先说明当前理解

推荐结构：

1. 当前理解
2. 当前状态
3. 下一步动作

这样可以先把失败定位成“入口恢复问题”，避免用户误以为文件本身损坏。

### 原则 2：保持非技术表达

推荐表达：

- 路径格式不对
- 中文路径兼容有问题
- 先复制一份只用于分析的临时副本

避免表达：

- 库层编码异常
- 宽字符兼容失败
- UTF-16/ANSI 转换异常

### 原则 3：不夸大未完成能力

推荐表达：

- 如果宿主环境支持复制动作，我会先复制一份 ASCII 临时副本
- 我会先继续读取工作簿的 Sheet 列表
- 这一步属于入口恢复，不代表已经修复底层库

## 验收映射

### 场景 A：路径格式问题

验收点：

1. 明确说这更像路径格式问题，而不是文件内容问题
2. 明确改用 Windows 原生反斜杠路径重新打开
3. 明确后续回到工作簿读取流程

### 场景 B：中文路径兼容问题

验收点：

1. 明确说文件存在，不是文件找不到
2. 明确说失败点更像中文路径兼容问题
3. 明确先复制 ASCII 临时副本，再继续读取 Sheet 列表

## 对 `cases.md` / `acceptance-dialogues.md` 的要求

### orchestrator 层

需要新增：

- 首次打开失败更像路径格式问题的场景
- 文件存在但中文路径直接读取失败的场景

### table-processing 层

需要新增：

- 路径格式问题的入口恢复场景
- 中文路径兼容失败后走 ASCII 临时副本的场景
- 单表入口先 `open_workbook` 再决定后续动作的约束

## 风险与边界

### 风险 1：Skill 说得对，但 Tool 还没有自动恢复动作

这意味着：

- 这轮先解决解释与路由问题
- 不夸大底层已经自动修复
- 后续还要把入口恢复能力真正下沉到 Tool 层

### 风险 2：Skill 误把中文路径失败说成文件不存在

这会导致：

- 用户误以为文件丢失
- 实际可恢复的问题被过早终止
- 会话状态被错误切走

### 风险 3：总入口和表处理层口径不一致

这会导致：

- 用户在 orchestrator 听到一种解释
- 到 table-processing 又变成另一种解释
- 整体体验不稳定

## 结论

入口恢复必须先于表处理、分析建模和决策助手的主流程。

本设计选择方案 B：

- 统一恢复顺序
- 同步到 orchestrator 与 table-processing 两层 Skill
- 保持 Skill 只做解释、路由和边界控制，不替代底层 Tool 的真实恢复实现
