# SheetMind Desktop GUI Design

**日期：** 2026-03-29  
**状态：** 已确认，可进入实施计划  
**适用范围：** `SheetMind` 首发本地桌面 GUI  

---

## 1. 目标

本设计文档定义 `SheetMind` 首发桌面 GUI 的产品结构、页面布局、视觉方向与交互原则。

本轮设计坚持以下前提：

- 终端用户首发版必须是本地桌面产品。
- 继续坚持 `Rust / exe / SQLite` 主线，不引入 Python 作为产品运行时依赖。
- GUI 是新增产品层，不重构现有 `stdin JSON -> Tool dispatcher -> stdout JSON` 引擎主链。
- GUI 首发目标是让终端用户能稳定完成 `导入 -> 确认表 -> 处理 -> 分析 -> 导出`。
- 后续必须预留 AI 接入口，但第一版不让 AI 反客为主。

---

## 2. 产品定位

`SheetMind` 不是聊天壳，也不是一堆 Excel 操作按钮的集合。  
它应当是一个面向终端用户的本地分析工作台，把现有 Rust 工具能力整理成一条可复用、可交付、可售卖的桌面产品流程。

用户真正需要解决的是：

- Excel 文件能否快速转成可分析的数据集。
- 不会 Python 时，能否独立完成常见分析任务。
- 分析完成后，能否直接导出可交付结果。
- 在功能逐渐增多时，界面是否仍然清楚、稳定、可长期办公使用。

---

## 3. 设计原则

### 3.1 架构原则

- GUI 只新增产品层，不推翻现有 CLI/Tool 外部协议。
- 继续以 `ToolRequest / ToolResponse / dispatcher::dispatch()` 为稳定能力边界。
- GUI 中所有任务动作，本质上都映射到现有或后续新增的 Rust Tool。
- 后续 AI 入口也必须复用同一套 Tool Contract，而不是绕过主链另起炉灶。

### 3.2 产品原则

- 先做终端用户真正能用的桌面主线。
- 先做顺手，再做炫技。
- 优先让用户知道“下一步做什么”。
- 隐藏技术名词，用业务语言表达。
- 错误提示必须可恢复，不能只报错。

### 3.3 视觉原则

- 整体气质：专业、稳定、可信、本地工具化。
- 结构走“专业工具台”，不是轻量玩具化小工具。
- 观感借一点现代精致感，但不过度追求花哨。
- 信息密度高于普通营销页，但不能压迫。

---

## 4. 方案结论

本轮采用混合方案：

- 外层骨架：经典专业工作台
- 内层体验：任务流工作台

选择该方案的原因：

- 能承接复杂功能与长流程。
- 最适合已有 Rust Tool Engine 的能力组织方式。
- 既适合终端用户首发，也适合后续高阶功能扩展。
- 可以稳定预留 AI 接入口，不必日后推翻 GUI 骨架。

---

## 5. 产品结构

对外产品统一命名为 `SheetMind`。  
内部架构分为两层：

- `SheetMind App`
  - 负责 GUI、项目管理、任务流、结果呈现、授权、导出。
- `SheetMind Engine`
  - 负责现有 Rust JSON Tool 引擎与执行主链。

用户侧不需要感知 `dispatcher`、`table_ref`、`ToolRequest` 等内部技术概念。

## 7. 总体布局

桌面端统一采用三栏式骨架：

```text
┌────────────────────────────────────────────────────────────────────┐
│ 顶栏：SheetMind / 当前项目 / 搜索 / 任务状态 / 授权状态 / 设置     │
├───────────────┬──────────────────────────────────────┬─────────────┤
│ 左侧主导航     │ 中间主工作区                           │ 右侧上下文区  │
│ 工作台         │ 当前任务核心内容                       │ 参数/说明/日志 │
│ 文件与表       │ 表格预览 / 图表 / 步骤卡 / 结果视图      │ 下一步建议     │
│ 数据处理       │                                        │             │
│ 分析建模       │                                        │             │
│ 报告导出       │                                        │             │
│ AI助手         │                                        │             │
│ 授权与设置     │                                        │             │
└───────────────┴──────────────────────────────────────┴─────────────┘
```

布局职责：

- 左侧主导航：稳定定位，减少迷路。
- 中间主工作区：承载当前主任务。
- 右侧上下文区：承载参数、说明、日志、下一步建议。

## 8. 页面设计

## 8.1 工作台

### 页面目标

用户启动后立即知道今天能做什么、从哪里开始、如何继续上次工作。

### 页面结构

- 顶部定位文案
- 最近项目卡片区
- 快速开始区
- 推荐任务模板区
- 最近导出与结果区

### 核心入口

- 打开 Excel 文件
- 继续上次工作
- 新建分析任务
- 查看报告与导出

### 推荐任务模板

- 单表清洗
- 多表合并
- 统计概览
- 异常诊断
- 回归分析
- 一键报告

### 设计意图

工作台不是营销首页，而是“进入工作”的起点。

---

## 8.2 文件与表

### 页面目标

将原始 Excel 文件转成“已确认、可分析”的数据集。

### 四步任务流

1. 选择文件
2. 选择工作表
3. 识别表区域
4. 确认表头并建立数据集

### 布局

- 左侧：文件树、Sheet 列表、已识别表列表
- 中间：Excel 预览区
- 右侧：识别结果、字段结构、确认操作

### 交互重点

- 自动高亮疑似表区域
- 支持多个候选区域切换
- 支持用户点选“这一行是表头”
- 对歧义字段给出“待确认”状态
- 确认成功后生成“当前数据集”

### 文案原则

- `schema` 显示为“字段结构”
- `header inference` 显示为“表头识别”
- `table_ref` 显示为“当前数据集”

### 错误态要求

- 识别失败时必须告诉用户原因
- 必须提供恢复路径：手动选择区域、切换 Sheet、查看示例

---

## 8.3 数据处理

### 页面目标

把高频表处理能力整理成可理解、可预览、可回退的工作面板。

### 功能分组

1. 列管理
2. 行处理
3. 缺失与类型
4. 派生计算
5. 聚合与透视
6. 多表操作

### 具体能力映射

#### 列管理

- 选择列
- 重命名列
- 文本规范化

#### 行处理

- 筛选
- 排序
- 去重
- Top N

#### 缺失与类型

- 填补缺失
- 类型转换
- 日期解析

#### 派生计算

- 派生字段
- lookup
- window calculation

#### 聚合与透视

- 分组汇总
- 透视表

#### 多表操作

- append
- join
- 推荐关联
- 推荐多表流程

### 页面布局

- 中上：功能分组卡片
- 中下：处理前后预览对比
- 右侧：参数区、说明区、影响提示

### 关键要求

- 点击功能后不要跳出主流程页
- 处理前后必须可对比
- 必须显示操作历史和回退入口

---

## 8.4 分析建模

### 页面目标

按用户任务目标组织分析入口，而不是按算法堆菜单。

### 任务导向入口

1. 了解数据整体情况
2. 检查数据质量问题
3. 查看字段之间关系
4. 查看趋势与变化
5. 做预测或分类
6. 让系统建议下一步

### 能力映射

#### 了解数据整体情况

- summarize
- stat summary

#### 检查数据质量问题

- diagnostics
- outlier
- distribution

#### 查看字段之间关系

- correlation

#### 查看趋势与变化

- trend analysis

#### 做预测或分类

- linear regression
- logistic regression
- cluster kmeans

#### 让系统建议下一步

- decision assistant

### 页面布局

- 上方：任务切换栏
- 中间左：参数配置
- 中间右：分析结果
- 下方：图表区
- 右侧：解释与风险说明

### 结果呈现原则

- 每个分析结果必须附一句人话总结
- 对模型类结果必须有风险提示
- 必须给出建议下一步动作

---

## 8.5 报告导出

### 页面目标

把一次分析整理成交付物，而不只是导出一个文件。

### 页面结构

- 左侧：导出模板列表
- 中间：报告预览
- 右侧：导出参数、路径、格式

### 导出模板

- 简版分析结果
- 诊断报告
- 完整业务交付包

### 页面能力

- 报告预览
- 图表输出
- Excel 报告输出
- CSV / 图片导出
- 输出路径确认
- 打开输出目录

### 产品意图

导出页是“交付中心”，不是单纯的“保存文件”。

---

## 8.6 AI 助手

### 页面定位

首版是辅助层，不是主入口。

### 首版职责

- 展示当前数据集上下文
- 推荐下一步分析
- 推荐可执行处理动作
- 对结果做简要解释

### 页面布局

- 左侧：当前上下文
- 中间：AI 建议卡片流
- 右侧：拟执行动作与确认

### 后续扩展口

未来扩展为：

`自然语言输入 -> 任务解析 -> Tool Plan -> Engine 执行 -> 结果回填`

---

## 8.7 授权与设置

### 页面目标

让用户顺利激活、理解状态、在出现问题时知道怎么恢复。

### 页面模块

- 授权状态
- 激活 / 刷新 / 解绑
- 本地设置
- 日志与版本信息

### 必须展示的信息

- 当前是否已授权
- 授权邮箱
- License 摘要
- 最近校验时间
- 当前设备状态

### 设计要求

- 少技术名词
- 少层级跳转
- 异常状态必须给出恢复路径

## 10. 视觉系统

## 10.1 视觉关键词

- 专业
- 稳定
- 高密度但不压迫
- 本地桌面工具感
- 现代精致但克制

## 10.2 配色建议

- 背景主色：`#F6F8FB`
- 主文本：`#1F2937`
- 次文本：`#64748B`
- 主强调：`#1E40AF`
- 辅助强调：`#3B82F6`
- 警示：`#F59E0B`
- 成功：`#16A34A`
- 错误：`#DC2626`

### 配色原则

- 主色承担结构强调，不承担全局大面积装饰。
- 警示色只用于待确认、风险、异常。
- 成功色只用于完成、有效状态。
- 不使用高饱和装饰色破坏专业感。

## 10.3 字体方向

不直接照搬英文代码风字体组合。  
中文桌面 GUI 优先考虑长期阅读舒适度与数字对齐表现。

排版建议：

- 页面标题：18-24
- 模块标题：14-16
- 正文：13-14
- 表格：12-13
- 辅助说明：12

## 12. 交互规则

- 一个页面只承载一个主任务。
- 参数配置优先在右侧展开，不频繁跳页。
- 所有会改动数据的操作都必须给预览和影响说明。
- 所有重要执行都显示进度状态。
- 所有结果都必须附一段人话总结。
- 所有错误都必须提供恢复路径。
- 所有导出都必须给出明确落点。

## 13. 空态、错态、成功态

## 13.1 空态

- 没有项目：引导“打开 Excel 文件”
- 没有数据集：引导“先确认一个表”
- 没有分析结果：引导“从基础概览开始”
- 没有导出记录：引导“生成第一份报告”

## 13.2 错态

### 文件错误

- 文件不存在
- 文件被占用
- 格式不支持

### 结构错误

- 没找到有效表区域
- 表头不明确

### 参数错误

- 列不存在
- 类型不兼容

### 执行错误

- 任务运行失败
- 导出失败
- 授权校验失败

## 13.3 成功态

- 数据集创建成功
- 分析完成并附结论摘要
- 导出完成并提供打开目录按钮
- 授权成功并显示有效状态

## 15. 技术落地约束

为了避免再次陷入重构，本轮 GUI 落地必须遵守：

- 现有 CLI 二进制入口继续保留，不被 GUI 替代。
- GUI 与 Engine 的能力边界必须清楚。
- GUI 首版优先调用已有 Tool Contract；确有必要时，只允许补桥接层，不允许改写业务核心。
- 授权沿现有 Lemon + SQLite 主线复用。
- 会话状态、结果引用、导出路径等，优先复用现有 runtime 目录体系。

---

## 16. 本轮结论

`SheetMind` 首发桌面 GUI 应当是一套“专业工作台式的本地分析桌面应用”：

- 外层结构稳定
- 内层任务流清楚
- 面向终端用户可直接使用
- 不推翻现有 Rust Tool Engine
- 为后续 AI 接入预留清晰接口
