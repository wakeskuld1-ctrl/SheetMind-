# Foundation Retrieval Evidence Tie-Break Design

## 目标

在当前 `RetrievalEngine` 已经具备：

- 文本主分数
- `source_ref` 次级优先级

的前提下，继续补一层更细的证据侧排序信号，让同分、同来源层级的节点可以优先返回：

1. 证据引用更多的节点
2. 定位更具体的节点

本轮只做 foundation 内部检索排序增强，不改外部合同。

## 本轮范围

本轮要做：

1. 新增证据数量 tie-break
2. 新增 locator 精度 tie-break
3. 把排序顺序冻结为：
   - 文本分数
   - `source_ref` 优先级
   - `evidence_refs` 数量
   - locator 精度
   - `node_id`

本轮不做：

- 修改 `RetrievalHit`
- 引入 `RetrievalConfig`
- 接入 CLI / Tool / GUI
- 引入向量检索、provider 或模型 rerank
- 改写命中条件

## 当前问题

Task 11 第二层完成后，retrieval 已经能在同文本分数时优先返回更像主数据源的节点，但仍有两个空缺：

1. 同分、同来源层级时，结果还会继续退回到 `node_id`
2. 节点虽然都属于同一来源层级，但证据丰富度和证据定位精度无法继续表达

这会让 foundation 检索结果在“都算对，但哪个更适合先看”这个层面仍不够稳定。

## 方案比较

### 方案 A：只看证据数量

做法：

- 同分时优先 `evidence_refs.len()` 更大的节点

优点：

- 最简单
- 风险最低

缺点：

- “证据多”不一定“证据更好”
- 无法区分一个单点定位和一个很粗的大范围定位

### 方案 B：只看 locator 精度

做法：

- 同分时优先 locator 更具体的节点

优点：

- 更贴近“证据质量”
- 对后续 evidence assembly 更有解释性

缺点：

- 证据数量信号被浪费
- 需要先定义 locator 精度规则

### 方案 C：双层证据排序

做法：

- 同分时先比证据数量
- 证据数量也相同时再比 locator 精度

优点：

- 信息利用更充分
- 仍然保持小步增强
- 与当前 retrieval 的分层 tie-break 设计风格一致

缺点：

- 比 A/B 稍复杂
- 需要补更多测试 fixture

## 选型

采用方案 C。

原因：

- 用户已批准继续按方案 C 推进
- 它仍然是 tie-break，不会破坏“文本分数优先、来源优先次之”的主轴
- 它能把证据数量和定位精度都纳入排序，而不需要提前引入配置系统

## 规则设计

### 第一层：证据数量

规则：

- `evidence_refs.len()` 越大，优先级越高

解释：

- 这里只作为次级证据充分性信号
- 不代表“证据越多越真”，只代表“可回溯证据更丰富”

### 第二层：locator 精度

规则目标：

- 更具体的定位优先于更宽泛的定位

本轮采用最小可解释规则：

1. 单点 locator 最优，例如 `A1`
2. 有界范围 locator 次之，例如 `A1:B3`
3. 无法识别为点或范围的 locator 作为兜底层

对范围 locator：

- 范围越小，优先级越高

对无法识别的 locator：

- 只作为最低层，不额外猜复杂语义

### 聚合口径

节点可能挂多个 `EvidenceRef`，因此需要固定聚合方式。

本轮采用：

- 数量层：直接使用 `evidence_refs.len()`
- 精度层：取节点内部“最具体”的 locator 作为该节点的 locator 精度代表值

原因：

- 这样既保留“证据多”的信号
- 又能避免某个节点挂了多个粗定位引用时完全掩盖掉其中一个高精度引用

## 边界控制

必须保持以下边界：

1. 文本分数仍然是第一优先级
2. `source_ref` 优先级仍然高于证据数量与 locator 精度
3. 证据侧 tie-break 不能参与“是否命中”的判断
4. 不修改 `RetrievalHit`
5. 不新增 `RetrievalConfig`

## 测试策略

### 测试 1：同分同来源时证据数量更多优先

验证：

- 两个节点文本分数相同
- `source_ref` 层级相同
- 一个节点有 2 条证据，另一个只有 1 条
- 结果优先返回证据更多的节点

### 测试 2：同分同来源同证据数时 locator 更具体优先

验证：

- 两个节点文本分数相同
- `source_ref` 层级相同
- `evidence_refs.len()` 相同
- `A1` 应优先于 `A1:D20`

### 测试 3：更高来源优先级不能被更多证据反压

验证：

- 一个节点来源更优，但证据较少
- 另一个节点来源较弱，但证据更多
- 结果仍然优先来源更优的节点

## 完成标准

满足以下条件即可视为本轮完成：

- 新增证据侧 tie-break 红灯测试
- 测试先红后绿
- retrieval 排序顺序扩展为：
  - 文本分数
  - 来源优先级
  - 证据数量
  - locator 精度
  - `node_id`
- foundation 最小回归集继续通过
