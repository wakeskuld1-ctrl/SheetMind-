# Warning Scenario Turning Point Design

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** 把老板报告的 `04_经营预警` 与 `05_未来场景预测` 串成“预警触发 -> 情景分化 -> 动作改变拐点”的完整时间主线，并同时输出利润拐点、毛利率拐点、灯色拐点、动作拐点和老板主拐点。

**Architecture:** 继续复用当前 Python 交付层的可解释预测逻辑，不引入新的黑盒模型。通过扩展 `WarningSignal` / `ScenarioForecast` 的交付字段和写表逻辑，在 `04/05` 页补齐按周期演化表、多拐点标记和老板主拐点口径，同时保持现有测试与图表合同兼容。

**Tech Stack:** Python, pytest, openpyxl, xlsxwriter

---

## 设计目标

1. `04_经营预警` 不再只是静态预警列表，而是能展示未来 3-5 个周期如何恶化或修复。
2. `05_未来场景预测` 不再只给单个“拐点月份”，而是能拆开解释：
   - 利润拐点
   - 毛利率拐点
   - 灯色拐点
   - 动作拐点
   - 老板主拐点
3. 老板可以看到“为什么这个月是主拐点”，团队也能看到“哪个动作先见效、哪个灯先变化”。
4. 规则必须可解释，可写进 Skill，不做黑盒。

## 方案对比

### 方案A：单一综合拐点

优点：
- 页面最简洁。
- 老板最容易一眼看懂。

缺点：
- 解释性太弱。
- 很难区分利润、灯色、动作谁先变化。

### 方案B：多拐点并列 + 老板主拐点

优点：
- 既能解释过程，又能给老板一个拍板锚点。
- 与用户“全部都要”的要求最一致。

缺点：
- 页面信息密度更高，排版要更克制。

### 方案C：只取最晚拐点

优点：
- 风控口径最保守。

缺点：
- 太硬，不利于经营讨论。
- 会压掉已经出现的改善信号。

## 选型结论

采用 `方案B`。

原因：
- 用户已经明确要求“全部把”，即利润、毛利率、灯色、动作都纳入拐点体系。
- 老板需要一个主拐点，但经营分析需要看到主拐点背后的构成过程。

## 拐点口径定义

### 1. 利润拐点

满足以下条件时成立：

- 当前周期利润止跌。
- 后续连续 2 个周期利润改善。

### 2. 毛利率拐点

满足以下条件时成立：

- 当前周期毛利率不再恶化。
- 后续连续 2 个周期毛利率改善。

### 3. 灯色拐点

满足以下条件时成立：

- 红灯转黄灯，或黄灯转绿灯。
- 变化必须可由已有信号解释。

### 4. 动作拐点

满足以下条件时成立：

- 动作已经进入见效期。
- 累计改善额开始覆盖动作成本或规模损失。

### 5. 老板主拐点

满足以下条件时成立：

- 利润拐点已出现。
- 毛利率拐点已出现。
- 灯色至少完成一次改善。
- 动作已经进入见效期。

老板主拐点是老板拍板的最终判断锚点，不替代四类基础拐点。

## 算法口径

### 趋势基线

- 使用最近 3 个周期加权移动平均。
- 收入、利润、毛利率分别计算基线趋势。

### 情景演化

- 情景A：继续当前策略
- 情景B：局部止损
- 情景C：结构优化

每个情景继续使用“基线趋势 + 动作改善斜率”的可解释口径。

### 灯色预测

继续使用规则引擎：

- 风险城市扩散
- 重点组合利润质量
- 周度预警信号
- 连续改善情况

### 主拐点推导

在每个情景中按周期滚动检查：

1. 利润拐点是否出现
2. 毛利率拐点是否出现
3. 灯色是否改善
4. 动作是否见效

四项条件首次同时满足的周期，即老板主拐点。

- 毛利率拐点
- 灯色拐点
- 动作拐点

- 周期
- 收入
- 利润
- 毛利率
- 灯色
- 动作状态

拐点标记支持：

- `利润拐点`
- `毛利率拐点`
- `灯色拐点`
- `动作拐点`
- `主拐点`

## 测试边界

至少锁定以下合同：

1. `04_经营预警` 出现：
   - `预警时间轴`
   - `预计灯色变化`
   - `预计主拐点`
2. `05_未来场景预测` 出现：
   - `利润拐点`
   - `毛利率拐点`
   - `灯色拐点`
   - `动作拐点`
   - `老板主拐点`
3. 情景表里继续保留：
   - `预测数据`
   - `拐点标记`
4. 真实 Excel 回读时能看到多拐点文本与关键情景对象。
