# Betting Optimizer Boundary Regression Design

## 背景
- 当前投注优化器已经修复了真实客户样例中的 `NoFeasibleSolution` 误判。
- 但用户进一步要求：不能只靠单一客户样例证明稳定性，而要主动构造边界值场景，自己写、自己测、自己发现问题。
- 本轮目标不是新增业务功能，而是补齐求解器和交付链路的边界回归覆盖，尽量提前暴露“极端分布下的最优性、可解性、状态表达和工作簿输出”问题。

## 目标
- 给求解器增加一组高信息密度边界测试，覆盖极端投注分布、极端目标值、零值稀疏分布和最优性边界。
- 给 CLI / workbook 交付链路补充少量端到端回归，确认极端输入不会只在单测层通过、但落到交付层又出问题。
- 如果边界测试暴露算法或桥接缺陷，本轮直接修复并补回归。

## 已确认业务契约
- 只能下调下注额，不能上调。
- 必须保持整数。
- 盈亏额 `> 0` 代表该号码亏钱、属于风险号码。
- 赚钱号码没必要退钱，但当前求解器为了达到全局最优，允许对任意正下注且未锁定的号码做向下搜索；最终是否退款由目标约束和最优性比较决定。
- 当前认可的优化目标顺序：
  1. 先尽量让亏损号码数接近目标值；
  2. 在同等接近度下，保留总下注额最大，也就是总退款最少；
  3. 如仍并列，取亏损号码数更小的方案。

## 方案比较

### 方案A：只补求解器单测
- 核心思路：
  - 只在 `tests/betting_optimizer_unit.rs` 中补人工边界样例。
- 优点：
  - 快，定位清楚。
- 缺点：
  - 不能覆盖 CLI 和 workbook 交付层。
- 风险：
  - 可能出现“单测没问题、交付层有问题”的漏检。

### 方案B：求解器边界单测 + 少量端到端回归
- 核心思路：
  - 在求解器单测层补主边界样例；
  - 再补 2~3 条 CLI / workbook 端到端验证。
- 优点：
  - 既能快速定位算法问题，又能覆盖客户真实交付链路。
- 缺点：
  - 比纯单测更重，执行时间更长。
- 风险：
  - 如果样例设计不够克制，测试会变慢且难维护。

### 方案C：随机压测为主
- 核心思路：
  - 主要用随机生成输入做批量回归。
- 优点：
  - 能更广泛扫边界。
- 缺点：
  - 断言难写，失败后定位成本高。
- 风险：
  - 容易得到“发现异常但原因不清”的噪声结果。

## 批准方案
- 采用方案B。

## 测试设计

### 一、求解器单测边界
- 单边投注：
  - 所有投注集中在一个号码上。
  - 关注点：是否会直接把该号码下调到满足最大亏损限制，且结果仍保持整数与不增约束。
- 完全均分：
  - 49 个号码平均分布。
  - 关注点：当前亏损号数量、是否在目标已满足时返回零退款，或在目标不满足时给出稳定最优解。
- 亏损号码少于 19：
  - 构造近似平均但只让少数号码略高于阈值。
  - 关注点：求解器无法靠“下调”把亏损号变多时，是否返回 gap 最小的方案，而不是错误宣称无解。
- 亏损号码远大于 29 且剩余号码为 0：
  - 构造稀疏极端分布。
  - 关注点：大量零值场景下，是否还能正确搜索，并保持结果稳定。
- 已经满足目标：
  - 最大亏损已经不高于阈值，且亏损号数已达目标。
  - 关注点：是否返回总退款 0。
- 不可行下界：
  - 即使把所有可调整下注降到 0，仍不能满足目标。
  - 关注点：是否明确返回 `NoFeasibleSolution`，而不是给错误可行解。
- 小规模最优性对拍：
  - 用 4~6 个号码的小样例做穷举，对比求解器结果和穷举真最优。
  - 关注点：避免再出现“最先解，不是最优解”的回归。

### 二、CLI / Workbook 端到端回归
- 单边极端样例：
  - 从 workbook 合同走完整 `solve` 链路，确认日志、输出文件和结果页状态正常。
- 已满足目标样例：
  - 端到端验证“零退款”不会被交付层误写成失败或空结果。
- 极端稀疏样例：
  - 多个 0 值 + 少数大额号码。
  - 验证模板读取、结果写回、日志与状态区一致。

## 验收标准
- 新增边界单测能稳定复现预期行为。
- 如发现缺陷，必须先看到红测，再修，再转绿。
- `betting_optimizer_unit` 全绿。
- `betting_solver_cli`、`betting_workbook_bridge_cli` 回归全绿，允许保留现有已知 `ignored` 用例。
- 若交付件需要更新，必须重建并替换客户包中的 `betting_solver.exe`。

## 风险
- 当前“目标优先、退款次优先”的比较器是本轮实现口径；若业务优先级以后再改，边界测试需要同步更新断言。
- 部分极端样例如果断言写得太死，会让未来合理重构变得困难，因此优先断言业务不变量与最优性关键指标，而不是每个号码的具体分配细节。

## 用户纠正记忆
- 错误类型：
  - 把最优解误做成最先解。
- 触发条件：
  - 搜索命中第一个可行候选后提前停止。
- 正确做法：
  - 必须按批准口径比较全局候选，不能提前停。
- 后续避免方式：
  - 增加小规模穷举对拍和多组极端边界回归，专门防这类回归。
