# Diagnostics Report Summary Handoff Design

## 背景

当前 `diagnostics_report_excel_report` 已经完成两轮交付增强：

- 第一轮把 `diagnostics_report` 封装成固定 4 页的 workbook 交付
- 第二轮把 `执行摘要` 升级为带管理字段的 5 页交付，并补上 `图表摘要`
- 第三轮继续打磨 `图表摘要` 页，补齐分布图和异常 Top 图

到这一步，图表页已经够用，真正还差的一块不再是“看见什么”，而是“诊断完以后下一步怎么接”。  
用户这轮明确选择的是“分析承接型”，不是老板拍板型，因此 `执行摘要` 不应继续往老板判断口径扩，而应升级成“诊断结果如何承接到复核、补数、建模”的入口页。

## 目标

在不改变 `diagnostics_report_excel_report` 主入口、不新增 Tool、不修改底层统计算法的前提下，把 `执行摘要` 从“字段摘要”增强为“分析承接摘要”。

本轮增强后，`执行摘要` 至少要回答下面几件事：

1. 是否建议继续复核
2. 是否建议继续补数
3. 是否建议进入建模
4. 当前主要阻塞项是什么
5. 下一步优先工具和动作是什么
6. 若要进入下一步，还缺什么条件

## 非目标

- 不新增新 Tool
- 不重构 `diagnostics_report`
- 不修改 `dispatcher / catalog / workbook draft` 主链
- 不引入第二套评分引擎
- 不把 `执行摘要` 做成完整流程系统或任务编排器

## 方案对比

### 方案 A：轻量承接字段

做法：

- 只在现有 `执行摘要` 中新增少量布尔/短句字段
- 例如 `是否建议继续复核 / 补数 / 建模`

优点：

- 改动最小
- 风险最低

缺点：

- 承接力偏弱
- AI 或人工接手时仍需继续读其它 sheet 才能确定下一步

### 方案 B：分析承接卡片

做法：

- 在现有 `执行摘要` 内新增一组固定承接字段
- 字段围绕“复核 / 补数 / 建模 / 阻塞项 / 下一步工具 / 准入条件”展开
- 所有判断继续复用 `diagnostics_result` 现有字段和请求参数做轻规则生成

优点：

- 最符合“分析承接型”目标
- 不需要改底层算法
- AI 接手时可以直接判断下一步怎么走

缺点：

- 规则比方案 A 多
- 需要补几条明确测试把判断口径锁住

### 方案 C：承接矩阵

做法：

- 把 `执行摘要` 做成 `复核 / 补数 / 建模` 三条 readiness 矩阵
- 每条给出状态、原因、建议

优点：

- 结构最清晰
- 后续扩展空间大

缺点：

- 比较重
- 容易把 `执行摘要` 做成第二个流程面板

本轮采用 **方案 B**。

## 设计原则

### 1. 继续留在同一个 Tool 内

所有承接增强继续留在 `diagnostics_report_excel_report` 内部完成，仍然由 `build_summary_dataframe()` 负责把摘要页组装出来。

### 2. 只复用已有结果，不反算新业务层

新的承接字段只能基于当前已有信息生成：

- `report_status`
- `warnings`
- `available_section_count`
- `recommended_actions`
- `key_findings`
- `sections`
- 请求里的 `correlation / outlier / distribution / trend`

不允许引入新的底层算法、不允许重新读取原始数据做第二次业务判断。

### 3. 规则轻、口径稳、便于解释

本轮承接字段应保持“轻规则、能解释、易测试”：

- 有 warning 或 report degraded，则优先引导复核
- 配置了 trend 且趋势 section 可用，说明更接近建模前状态
- section 可用数偏少，优先引导补数或复核
- `recommended_actions` 的首项继续作为下一步动作的主线输入

### 4. 摘要页仍然是数据页，不改 workbook 主结构

本轮只增强 `执行摘要` 的内容，不调整 workbook 页数，不新增新页。

## 承接字段设计

建议在现有字段之后新增以下固定字段：

- `复核建议`
- `补数建议`
- `建模建议`
- `建议优先工具`
- `建议目标字段`
- `建议时间字段`
- `当前主要阻塞项`
- `进入下一步前需满足条件`

## 字段口径

### 1. 复核建议

用于回答“现在是否应该先回头看数据或配置”。

建议口径：

- `建议立即复核`：`report_status = degraded` 或存在 warnings
- `建议重点复核`：有较多 `key_findings`
- `可进入下一步前抽样复核`：状态良好且 warnings 为空

### 2. 补数建议

用于回答“数据是否已经足够支撑后续分析”。

建议口径：

- `建议优先补数`：可用 section 偏少，或某个关键 section 不可用
- `建议按需补数`：主 section 大多可用，但仍有局部缺口
- `当前可不优先补数`：主 section 已齐

### 3. 建模建议

用于回答“是否适合继续进入模型或更深分析”。

建议口径：

- `暂不建议进入建模`：degraded 或 warnings 不为空
- `建议先复核后再评估建模`：section 基本可用但仍有阻塞
- `可以进入下一步建模准备`：状态良好且趋势/目标相关配置明确

### 4. 建议优先工具

用于直接告诉 AI 或人工下一步优先调用哪个 Tool。

本轮建议限制在已有主线内：

- `diagnostics_report`
- `diagnostics_report_excel_report`
- 后续建模链对应的现有统计/建模 Tool 名称

本轮不做复杂路由，只给出一个最优先工具字符串。

### 5. 建议目标字段 / 建议时间字段

优先直接从请求中读取：

- `correlation.target_column`
- `trend.time_column`
- `trend.value_column`

如果没有，则给出 `未配置` 或 `建议补充` 口径。

### 6. 当前主要阻塞项

建议从以下来源按优先顺序生成：

1. warnings 首项
2. 缺失的关键 section
3. 缺少目标字段或时间字段

### 7. 进入下一步前需满足条件

这是对阻塞项的短句式落地，例如：

- `先补齐趋势时间字段后再进入建模`
- `先处理 warning 后再继续深度分析`
- `先补齐异常或分布观察后再做建模判断`

## 降级规则

- 只要 workbook 仍能生成，就继续生成这些承接字段
- 若承接判断缺少足够信息，则输出保守值，不报错
- 保守值优先写“建议复核”或“建议补数”，而不是强行推荐建模

## 测试策略

本轮至少补 3 类测试：

1. 正常场景下，`sharedStrings.xml` 包含新增承接字段
2. 降级场景下，承接判断倾向复核而非建模
3. 配置齐全场景下，能写出 `建议目标字段 / 建议时间字段`

必要时再补：

4. 未配置 trend 时，`建议时间字段` 走保守口径
5. warnings 为空但 section 不齐时，`补数建议` 仍生效

## 下一步

实施时继续保持：

- 先写红测
- 先确认失败点是“承接字段还不存在”
- 再做最小实现
- 最后跑专项测试和全量 `cargo test`
