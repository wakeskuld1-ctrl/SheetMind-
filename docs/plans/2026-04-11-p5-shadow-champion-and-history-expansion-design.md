# P5 Shadow Champion And History Expansion Design

## 背景

`P4` 已经把证券主链推进到“可回填、可训练、可聚合、可主席消费”的阶段：

- 历史外部代理可以治理化回填
- `direction / return / drawdown / path_quality / upside_first / stop_first` 已有正式训练入口
- `master_scorecard` 与 `chair_resolution` 已能消费多头摘要

但当前正式产物仍然主要停在 `candidate` 级别。系统还缺 3 个关键能力：

1. 缺少 `candidate -> shadow -> champion` 的正式晋级制度
2. 缺少真实历史代理的持续扩容与覆盖评估
3. 缺少审批链只消费合格模型等级的硬约束

## 目标

把当前“研究级 candidate 链”升级为“可治理的 shadow/champion 链”，并同时把真实历史代理扩容纳入正式治理，使后续证券分析不只是“能训练”，而是“知道哪些模型允许进入审批主链”。

## 方案选型

### 方案 A：先做 shadow/champion，后补历史代理

优点：
- 最快建立模型晋级制度
- 审批安全边界会先提升

缺点：
- 很多 ETF 子池仍然会长期被手动代理或占位字段卡住
- shadow/champion 会建立在偏薄的数据底座上

### 方案 B：先做历史代理扩容，后做 shadow/champion

优点：
- 数据底座先变厚
- 后续晋级判断更接近真实实盘

缺点：
- 短期内缺少正式“允许放行/必须降级”的治理规则
- 审批链依然要继续消费 candidate 结果

### 方案 C：一阶段内一起做，但按顺序推进

顺序：
1. 先做真实历史代理扩容治理
2. 再做 `candidate -> shadow -> champion` 晋级模型
3. 最后让审批链只消费合格等级模型

优点：
- 数据与治理一体收口
- 后续不会反复返工审批口径
- 做完即可形成“实盘辅助雏形”

缺点：
- 改动面较大
- 需要严格按阶段推进，避免边界蔓延

## 推荐方案

推荐 **方案 C**。

原因：
- 当前最大风险不再是“没有训练”，而是“训练结果没有正式等级”
- 如果先做晋级制度但不补历史扩容，制度会长期卡在形式上
- 如果先做扩容但不做晋级制度，审批链仍然无法真正区分“研究候选”和“可托付模型”

## 设计范围

### 范围内

- 新增真实历史代理扩容治理对象与回填运行记录
- 新增 `shadow` 级模型状态与评估记录
- 新增 `champion` 晋级判断和注册状态
- 让审批链与 `decision package` 读取模型等级并做硬约束
- 补齐对应 CLI 合同测试、审批回归和模型治理测试

### 范围外

- 不在这一阶段接入新的实时行情抓取
- 不在这一阶段做全新的复杂回归算法替换
- 不在这一阶段重构投委会席位体系
- 不在这一阶段做 UI 层展示改版

## 核心设计

### 1. 历史代理扩容治理

新增一类正式运行记录，描述某一批真实历史代理扩容任务：

- 扩了哪些 `instrument_scope / instrument_subscope`
- 覆盖了哪些字段
- 覆盖了哪些时间窗口
- 回填后哪些 horizon/路径事件覆盖率提升了

目标不是立刻做抓取引擎，而是把“历史代理是否足够支撑晋级”变成可审计事实。

### 2. 模型等级治理

正式把模型分成三层：

- `candidate`
- `shadow`
- `champion`

含义：

- `candidate`
  - 可训练、可测试、可供研究使用
  - 不应直接被当作正式审批放行依据

- `shadow`
  - 已经过基本拟合、覆盖和稳定性门槛
  - 可以并行跟踪真实结果
  - 可进入审批链作为“参考量化上下文”

- `champion`
  - 已满足更严格的覆盖、稳定性、回放和治理门槛
  - 才允许被审批链当成正式放行量化依据

### 3. 晋级判断

晋级不只看单一 `accuracy/AUC`，而是组合判断：

- 样本量状态
- 类别平衡状态
- 路径事件覆盖状态
- 多 horizon 可用性
- 历史代理覆盖状态
- 是否已有 shadow 运行记录

这一步仍保持规则透明，不做黑盒晋级器。

### 4. 审批链消费规则

审批链需要区分三种情形：

- `champion`
  - 可正常消费模型结论

- `shadow`
  - 可以写入审批摘要，但必须显式标注“影子模型参考，不得单独放行”

- `candidate / unavailable`
  - 只能作为研究或治理阻断信息
  - 不得被表述成正式量化放行

### 5. 文档与复盘闭环

这一阶段的每次扩容、晋级和降级，都必须进入：

- 模型注册表
- refit / shadow / promotion 运行记录
- 审批摘要或决策包
- 任务日志

这样后面复盘时能知道：
- 模型为什么被升格
- 为什么被拒绝升格
- 哪些真实历史代理缺口导致它只能停在 `candidate`

## 测试策略

### 单测

- 历史代理扩容记录对象构建
- 模型等级判定
- 审批链等级闸门

### CLI 回归

- 历史代理扩容 CLI
- shadow 评估 CLI
- champion 晋级 CLI
- submit_approval 等级消费回归

### 相邻回归

- `security_scorecard_training_cli`
- `security_master_scorecard_cli`
- `security_chair_resolution_cli`
- `security_decision_submit_approval_cli`

## 成功标准

完成后，系统应具备这些能力：

- 能正式记录“哪些真实历史代理已经补齐”
- 能正式判定模型处于 `candidate / shadow / champion`
- 审批链不再把所有训练产物一视同仁
- 持仓、审批、决策包里能清楚看到模型等级语义

## 风险

- 若历史代理扩容仍过薄，很多模型会长期卡在 `candidate`
- 若晋级门槛设计过松，会把 research candidate 误当 champion
- 若晋级门槛设计过严，系统会长期停在“治理很好但永远不放行”

## 下一步

按推荐方案 C 进入实现计划，拆成：

1. 历史代理扩容治理对象与 CLI
2. shadow 运行与评估对象
3. champion 晋级与注册状态
4. submit_approval / decision package 的等级消费
