## 1. 文档目的

这份文档不是零散想法汇总，而是当前仓库后续阶段的正式路线说明。

它要解决 4 个问题：

- 现在这个平台做到哪了，为什么还不能算闭环。
- 后续优先级应该先做什么，为什么不是继续深挖投决会。
- 用户提出的 7 个方向应该怎么并入正式主线。
- 在“不用大模型、不用 Token、尽量依赖免费数据”的约束下，怎样推进到可落地版本。

## 2. 当前状态判断

当前正式主链已经具备：

- 投前分析：`technical_consultation_basic`、`security_analysis_contextual`、`security_analysis_fullstack`、`security_analysis_resonance`
- 统一决策入口：`security_decision_briefing`
- 正式投决入口：`security_committee_vote`
- ETF 最小分流
- `risk_breakdown` 强一致性门禁

但当前系统本质上仍然偏“投前分析平台”，还不是完整的闭环投研平台。

原因很直接：

- 有投前结论，但没有正式的赔率系统。
- 有执行阈值，但没有正式的仓位管理系统。
- 有历史研究雏形，但没有正式的投后复盘闭环。
- 有投决会结果，但没有把投前、投中、投后串成统一评估对象。

因此，当前缺的不是“再多一层投决会规则”，而是闭环中的后两段：

- 投中：怎么下、下多少、何时加减、错了怎么退。
- 投后：结果怎样、偏差在哪、策略是否失效、因子权重是否需要调整。

## 3. 目标定义

后续目标不是把平台做成“更会分析”，而是做成“能形成决策闭环”。

闭环最小定义如下：

- 投前：给出方向、赔率、风险、适用场景。
- 投中：给出仓位、分批、触发条件、回撤约束、退出纪律。
- 投后：记录执行、回看结果、评估偏差、修正模型。

只有同时具备这三段，平台才具备真正的策略深度。

## 4. 三种推进方案

### 方案 A：信息面优先

先集中补大盘、行业、公告、财报、资金流、北向、情绪、流动性，再回头做仓位和复盘。

优点：

- 报告会更“像人”，分析更全面。
- 用户体感提升快，单次分析内容更丰富。

缺点：

- 会长期停留在“说得更全”，但没有闭环。
- 数据源最容易卡住，尤其是免费深层数据。
- 仓位与复盘迟迟缺位，平台深度仍然不够。

### 方案 B：闭环优先

先补赔率系统、仓位管理、投后复盘，再逐步叠市场结构与深层信息面。

优点：

- 最快形成真正的闭环平台。
- 早期可以主要依赖已有价格、成交量、波动率、历史样本，不被外部数据源卡死。
- 后续接任何信息面数据，都能落到赔率、仓位和复盘层里，不会沦为“只会加描述”。

缺点：

- 报告层面的“信息很全”提升会慢一点。
- 前期需要更严格地定义赔率、仓位和复盘合同。

### 方案 C：全量并行

同时做赔率、仓位、复盘、市场结构、北向、情绪、流动性、预期差。

优点：

- 纸面上最完整。

缺点：

- 范围过大，最容易失控。
- 免费数据源、合同设计、验证体系会同时成为阻塞点。
- 当前仓库不适合这么并行推进。

### 推荐结论

推荐继续走方案 B，也就是“闭环优先”。

理由：

- 这是最符合平台深度要求的路线。
- 这是最不依赖外部复杂数据源的路线。
- 这是最容易在当前仓库正式主链上连续落地的路线。

## 5. 用户提出的 7 个方向如何并入主线

| 用户方向 | 是否纳入 | 推荐阶段 | 说明 |
|---|---|---|---|
| 技术面平衡计分卡 + 因子权重回归 | 纳入 | 第二优先级 | 适合作为赔率系统和复盘系统之间的桥梁 |
| 北向资金 / ETF 大单 / 深层资金流 | 纳入 | 后置增强 | 有价值，但最容易被免费数据源卡住 |
| 市场结构分析：牛熊、轮动、风格、宏观流动性 | 纳入 | 中优先级 | 应作为仓位和赔率的上层调节器 |
| 赔率系统：盈亏比、胜率、回报率、最大回撤 | 核心主线 | 第一优先级 | 当前最该补完的正式能力 |
| 仓位情况 | 核心主线 | 第一优先级 | 没有仓位系统，就不是闭环 |
| 预期差 / 市场共识 | 纳入 | 后置增强 | 免费数据较难，第一版宜做弱代理版 |
| 情绪周期 / 流动性优先级 | 纳入 | 中后优先级 | 可先做弱数据版，再做深层增强版 |

## 6. 正式主线调整

后续正式主线调整为：

1. 赔率系统
2. 仓位管理
3. 投后复盘
4. 市场结构层
5. 技术面平衡计分卡与周度因子权重
6. 深层信息面增强

不再把“继续扩投决会规则”作为最高优先级。

投决会后续只做必要收口，不再作为主线吞噬后续版本资源。

## 7. 阶段拆分

### 阶段 M3：闭环底座

#### M3-1 赔率系统

目标：

- 让平台对每次分析给出可量化的“值不值得做”。

最小输出建议：

- `win_rate_5d`
- `win_rate_10d`
- `expected_return_5d`
- `expected_return_10d`
- `expected_drawdown_5d`
- `expected_drawdown_10d`
- `payoff_ratio`
- `odds_grade`
- `evidence_strength`
- `research_limitations`

主要依赖：

- 已有价格历史
- 技术指标快照
- `signal_outcome_research`

#### M3-2 仓位管理系统

目标：

- 让分析结果变成可执行的仓位方案。

最小输出建议：

- `entry_mode`
- `starter_position_pct`
- `max_position_pct`
- `add_on_trigger`
- `reduce_on_trigger`
- `hard_stop_trigger`
- `liquidity_cap`
- `position_risk_grade`
- `execution_notes`

设计原则：

- 仓位不是单独拍脑袋给出，而是由赔率、波动率、流动性和市场状态共同决定。
- 仓位建议必须与 `committee_payload` 可对齐，但不依赖继续扩委员规则。

#### M3-3 投后复盘系统

目标：

- 让平台能对自己的建议负责。

最小输出建议：

- `suggestion_id`
- `snapshot_date`
- `entry_context`
- `planned_position`
- `actual_result_window`
- `max_drawdown_realized`
- `max_runup_realized`
- `thesis_status`
- `execution_deviation`
- `model_miss_reason`
- `next_adjustment_hint`

复盘维度：

- 方向对不对
- 仓位给得对不对
- 时机对不对
- 市场状态判断是否偏差
- 因子权重是否需要修正

### 阶段 M4：结构与增强

#### M4-1 市场结构层

目标：

- 把个券判断放进市场状态里解释，而不是只盯单票。

最小模块：

- 市场状态：牛市 / 熊市 / 震荡
- 风格状态：大盘 / 小盘 / 价值 / 成长
- 行业轮动状态
- 流动性状态
- 风险偏好状态

第一版允许使用弱代理：

- 指数趋势
- 成交额变化
- 涨跌家数
- ETF 溢价
- 波动率状态

#### M4-2 技术面平衡计分卡

目标：

- 把技术指标从“口头分析”升级成“可比较、可回归、可修正”的评分体系。

第一版建议：

- 趋势组
- 动量组
- 波动组
- 资金组
- 均值回归组
- 高阶确认组

输出：

- 单因子分
- 组内分
- 总分
- 周度权重版本
- 权重更新时间

#### M4-3 深层信息面增强

目标：

- 在闭环底座已经成立后，再给系统补深层解释力。

优先顺序建议：

1. 市场结构与流动性
2. 情绪周期
3. 北向资金 / ETF 资金流
4. 预期差 / 市场共识

## 8. 免费数据策略

当前约束是：

- 不使用大模型做抓数
- 不依赖 Token
- 尽量降低用户安装门槛

因此数据策略应分 3 层，而不是一步到位追求“最全”。

### 第一层：无额外依赖可落地层

优先使用仓库已有数据和可直接同步的数据：

- K 线价格
- 成交量
- 换手率
- 常规技术指标
- ETF 溢价
- 指数走势

这一层足够支撑：

- 赔率系统第一版
- 仓位系统第一版
- 投后复盘第一版

### 第二层：免费公开弱结构化层

适合后续逐步接入：

- 交易所公告
- 基金公告
- 指数成分信息
- 宏观公开统计
- 利率与流动性公开数据

这一层主要服务：

- 市场结构层
- 流动性优先级
- 弱版本预期差

### 第三层：免费但稳定性较弱的增强层

例如：

- 北向资金细分
- ETF 大单成交
- 更细颗粒资金流
- 市场热度与散户参与代理指标

这一层不应成为主链单点依赖，只能作为增强项。

## 9. 推荐的正式落地顺序

### 路线 1：先成环

1. 补完 `signal_outcome_research`
2. 新增赔率输出
3. 新增仓位管理输出
4. 新增投后复盘记录与总结

### 路线 2：再加状态

1. 新增市场结构层
2. 把市场状态并入赔率和仓位
3. 把结构状态并入 briefing / report

### 路线 3：最后做增强

1. 技术面平衡计分卡
2. 周度因子权重回归
3. 深层信息面接入
4. 情绪与预期差增强

## 10. 对现有模块的映射建议

| 目标层 | 优先挂接模块 | 说明 |
|---|---|---|
| 赔率系统 | `signal_outcome_research` | 这是现有主链最自然的扩展点 |
| 仓位管理 | `security_decision_briefing` | 作为统一决策入口最合适 |
| 投后复盘 | `runtime` + `signal_outcome_research` | 需要能记录建议与后续结果 |
| 市场结构 | `security_analysis_contextual` | 当前最适合承接市场、行业、风格层 |
| 平衡计分卡 | `technical_consultation_basic` + 研究层 | 当前技术指标已够做第一版评分 |
| 深层信息面 | 新增独立 adapter 层 | 不建议直接糊进现有主链 |

## 11. 阶段验收标准

### M3 完成标准

- 任一标的都能给出赔率摘要。
- 任一标的都能给出仓位建议，而不是只有方向判断。
- 任一建议都能在事后形成可追踪复盘记录。
- briefing 输出从“结论性建议”升级成“结论 + 赔率 + 仓位 + 复盘追踪入口”。

### M4 完成标准

- 个券分析会显式带市场结构解释。
- 技术面可以产出统一评分，而不只是自然语言描述。
- 深层信息面开始作为增强项进入，不再只是手工补充。

## 12. 明确不做的事

当前阶段不做以下事情：

- 不再把投决会规则继续无限扩展成更复杂的制度模拟。
- 不把 foundation 通用能力强行提前耦合进证券分析主链。
- 不把“数据看起来很多”误当成“平台已经闭环”。
- 不为深层信息面去引入高门槛运行环境。

## 13. 最终结论

当前仓库已经完成了“投前分析 + 统一 briefing + 正式投决会”的基础设施。

下一阶段最重要的，不是继续证明系统“会分析”，而是证明系统“能形成闭环”。

因此后续正式主线应明确收口为：

- 先做赔率系统
- 再做仓位管理
- 再做投后复盘
- 然后补市场结构
- 最后补深层信息面增强

这条路线更稳，也更符合当前仓库、当前数据条件和当前平台目标。
